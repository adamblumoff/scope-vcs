Executive Read

ADA-45 fixed the file-size and navigation problem. It did not make the database schema canonical. The next move should remove schema facts from every place except a single schema contract.

Current state

The DB module now has smaller files, but table shape is still repeated across schema_*.rs, metadata_schema/*, entities/*, domain structs, and mapping helpers. There are 44 Rust files under crates/scope-core/src/db.

Goal

Make table names, column names, SQL types, nullability, primary keys, indexes, foreign keys, drift inventory, and reset order come from one typed contract. Keep domain structs authoritative for behavior and invariants.

Non-goal

Do not build a migration framework, normalize commits into rows, or add a generator toolchain. This should be a deletion-oriented persistence cleanup for pre-alpha Scope.

Canonical Target

The canonical schema should be a small Rust contract, not another document. Every runtime path that needs table shape should ask this contract. 

db/
  schema_contract/
    mod.rs              // TableSpec, ColumnSpec, IndexSpec, ForeignKeySpec
    auth.rs             // users, auth_identities, cli auth/session tables
    repositories.rs     // repository row and repository fact tables
    collaboration.rs    // members and invites
    jobs.rs             // locks, reset events, outbox, cleanup jobs
    read_models.rs      // projection read model tables
  schema.rs             // creates, drifts, and resets from schema_contract
  rows/ or row_models/  // row translation only, no schema authority

Schema contract owns

  • Table names.
  • Column names.
  • SQL types.
  • Nullability.
  • Primary keys.
  • Indexes.
  • Foreign keys.
  • Drop/reset order.
  • Catalog-row drift flags.

Domain owns

  • Repository behavior.
  • Visibility rules.
  • Allowed transitions.
  • Side effects.
  • Authorization decisions.
  • Public API shape.

Rows own only

  • Persisted value decoding.
  • Persisted value encoding.
  • Lossy numeric bounds where unavoidable.
  • Joins into domain aggregate loaders.

Approaches

There are three realistic choices. The recommended path avoids both weak file shuffling and a custom code generator.

Recommended

Typed schema contract with generated DDL at runtime

Define typed table specs in Rust and make schema.rs produce SeaQuery create-table, index, drift, and reset behavior from those specs. Then remove metadata_schema/* and the hand-written schema_*.rs duplication.

  • Deletes real duplicated schema shape.
  • Keeps everything in the Rust compile graph.
  • Does not introduce build scripts or code generation.
  • Lets us migrate family by family without behavior change.
Fallback

Contract plus parity tests only

Add a schema contract and tests that compare it with the existing DDL and entity definitions, but leave the existing DDL and SeaORM entities as operational code.

  • Lower risk for the first commit.
  • Useful as a stepping stone.
  • Not enough as the final state because it adds one more representation.
Reject now

Generate SeaORM entities from a private DSL

A generator could make schema and entity structs mechanically consistent, but it creates a private toolchain before the product has enough usage pressure to justify it.

  • More machinery than the current dogfooding stage warrants.
  • Harder for future agents to inspect and edit safely.
  • Likely to become its own architecture project.

Recommended Work Order

Keep this as one implementation track with small commits. Stop when a representative column addition no longer crosses separate schema inventory and DDL definitions.

1

Add the schema contract types

Create db/schema_contract/mod.rs with typed structs for tables, columns, indexes, foreign keys, reset metadata, and drift behavior. Keep the API boring: simple arrays of static specs, no macros beyond tiny helpers if they remove repetition. 

pub struct TableSpec {
    pub name: &'static str,
    pub columns: &'static [ColumnSpec],
    pub primary_key: PrimaryKeySpec,
    pub indexes: &'static [IndexSpec],
    pub foreign_keys: &'static [ForeignKeySpec],
    pub counts_for_catalog_rows: bool,
    pub reset_order: u16,
}
2

Port one low-risk table family end to end

Start with read models or cleanup jobs. Generate the DDL and drift inventory from the contract for that family while leaving the rest of the schema untouched. This proves the contract shape before touching repository core tables.

3

Replace metadata schema inventory

Once all families have table specs, delete metadata_schema/*. metadata_schema_tables(), reset lists, catalog-row checks, and uniqueness tests should all read from schema_contract.

4

Replace hand-written schema builders

Convert schema_auth.rs, schema_cleanup.rs, schema_repositories.rs, and schema_collaboration.rs into generated create-table calls from the contract. Keep custom behavior only where a constraint cannot be represented cleanly; treat every exception as suspicious.

5

Demote SeaORM entities from schema authority

SeaORM entity structs still restate row fields. After DDL and drift come from the contract, replace entity modules family by family with row structs and query helpers that use contract table and column names. The row structs may repeat persisted fields for decoding, but they must not own table shape or constraints.

This is the only behavior-risky part. Do it after contract DDL is proven, and keep each family covered by existing DB tests.

6

Add a schema-string guard

Add a focused test or script that fails when a current table name or column name appears outside approved locations. This is not a style rule; it protects the canonical contract from slowly regrowing parallel inventories.

Deletion Map

The plan should delete code as it goes. If a step only adds a layer, it is unfinished.

Current surface Final role Deletion target
metadata_schema/* None after contract exists. Delete; drift and reset read from schema_contract.
schema_*.rs Thin orchestration or gone. Delete table-specific DDL builders after contract generates DDL.
entities/* Temporary row query adapter. Replace SeaORM schema-bearing entities with contract-backed row helpers.
repository_rows.rs Aggregate loading and saving. Keep behavior, but remove table/column literals and broad fact-row plumbing where the contract can drive query helpers.
domain/store.rs Domain data and behavior inputs. Keep. This is not schema authority; it is product authority.

Boundaries

Canonical schema work can get too ambitious quickly. Keep these constraints explicit.

Allowed

  • Destructive reset changes for pre-alpha local data.
  • Table-family-by-table-family replacement.
  • Mechanical query helper changes.
  • Tests that lock down generated DDL parity.

Not allowed

  • Backward-compatible migrations.
  • A custom code generator.
  • Commit graph normalization.
  • API, Git, auth, or worker behavior changes.

Pause if

  • A table-family conversion changes semantics.
  • The contract needs a mini language.
  • Tests require broad rewrites.
  • The diff grows without deleting old surfaces.

Verification

The core proof is not only that tests pass. The proof is that one table or column change has a smaller and more obvious edit path.

  • Run cargo fmt --all --check.
  • Run cargo clippy --workspace --all-targets -- -D warnings.
  • Run cargo test -p scope-core.
  • Run cargo check --workspace.
  • Run API and worker CI after pushing.
  • Test metadata reset on a fresh DB.
  • Test metadata drift detection from the contract.
  • Compare generated DDL against the current DDL before deleting builders.
  • Search for table and column literals outside approved contract and SQL sites.
  • Document a representative column addition before and after.
  • Run autoreview with the 15 minute timeout before opening the PR.

Exit Criteria

This work is done when a schema change is obviously smaller, not when the persistence layer is theoretically perfect.

Must be true
  • Adding a column changes the schema contract and row/domain mapping only.
  • DDL creation reads from the schema contract.
  • Drift detection reads from the schema contract.
  • Reset drop order reads from the schema contract.
  • No current table or column inventory exists outside the contract.
May remain
  • Domain structs remain separate because they express product behavior.
  • Row structs remain separate if they only decode and encode persistence values.
  • Query helper modules remain separate if they own workflow-specific persistence behavior.

Closeout rule: do not start another architecture phase. Either land the contract and delete duplicated surfaces, or stop and dogfood until a concrete schema change proves the next cut.
