Executive read

Scope has a strong domain center, but the persistence and read side still behave like a serialized catalog with large repository aggregates. The next architecture push should keep domain behavior authoritative while breaking storage, reads, and side effects into smaller durable facts.

Current recommendation: Phase 0 is complete enough to use as the baseline harness. Move next to ADA-34: remove normal authenticated request writes before touching the global metadata lock or repository JSON shape. Phase 1 is landed on main. Move next to ADA-35: replace the catalog-wide metadata lock with scoped transactional invariants before splitting repository JSON.

Phase 2 target: known Clerk users and valid CLI sessions should resolve principals through read-only database work. repo and cleanup mutations should stop coordinating through one global lock row. Use database constraints, scoped row locks, and typed cleanup/job rows so unrelated repos can move independently.

Current state

Recent work has cleaned up several boundaries: object-store modules are split, staged push rules moved into domain ownership, and collaboration HTTP glue was condensed. The remaining data problem is less about file size and more about the shape of state and side effects.

Strong

Domain model

StoredRepository, SourceGraph, visibility policy, staged updates, collaboration rules, and projections hold the important product rules in durable domain code.

Mixed

Persistence

Postgres has real tables for users, auth identities, members, invites, CLI auth, and repositories. Auth reads are cleaner after Phase 1, but broad metadata operations still coordinate through the same lock row. The repository table still stores graph, policy, tokens, pending import, snapshot, and staged update as JSON.

Costly

Reads

Repo detail, files, projection preview, history, and Git clone paths commonly rehydrate the full repository aggregate and replay projection logic on demand.

Hot

Side effects

Git subprocesses, object-store reads and writes, projection Git repo creation, bundle restore, and cleanup can run inline with request handling.

Typical hot path today

This is the pattern I would expect to dominate current latency and capacity. It is not wrong for pre-alpha, but it is the shape that will cap growth.

Auth
Clerk JWT is verified, then resolve_clerk_user can enter a metadata write transaction to create or update the Scope user snapshot. Established Clerk and CLI principals now resolve through read-oriented paths. Explicit lifecycle flows still reconcile user snapshots where stored auth state depends on them.
Metadata
Repo reads take the metadata lock row in share mode. Mutations and auth flows take it in update mode. repo lifecycle, cleanup, reset, and explicit auth lifecycle flows take it in update mode. Many unrelated repos therefore still coordinate through one row.
Aggregate
A repository row is loaded and decoded into StoredRepository, including graph, policy, token state, snapshots, pending import, and staged updates.
Compute
Projection, file tree, preview, and history are computed from the source graph on request. Public and owner views often repeat the same work.
Git and blobs
Git clone paths may build or restore bare repos using subprocesses and object-store reads. Object-store calls currently return whole byte buffers.

Pressure points

These are the places where the architecture is likely to bend first as data or concurrency grows.

Area Current behavior Why it matters Likely fix
Global metadata lock One scope_metadata_locks row protects broad metadata invariants. Unrelated users and repos can queue behind auth writes, repo mutations, cleanup updates, and large reads. Normal auth reads are no longer the issue, but unrelated repos can still queue behind repo mutations, cleanup updates, metadata reset, and explicit lifecycle writes. Use constraints, row-level locks, scoped repo locks, and typed job tables instead of one catalog lock. This is now the Phase 2 target.
Large repository JSON scope_repositories stores graph, policy, visibility events, tokens, snapshots, pending import, and staged update as JSON. Normal detail/list/settings reads become sensitive to repo history size and JSON decode cost. Split canonical facts into domain-owned tables and rehydrate full aggregate only for mutation paths.
Derived reads Projections, file trees, history, and clone artifacts are built from canonical graph data on demand. Read cost tracks total repo history instead of result size. Materialize read models keyed by repo revision, source, and audience class.
Inline Git work Git subprocesses and bundle/projection repo creation happen during request handling. A burst of pushes or clones can block normal UI traffic. Add outbox jobs, workers, prepared artifacts, concurrency limits, and explicit backpressure responses.
Object-store interface ObjectStore::get returns Vec<u8>; S3 uses blocking reqwest inside block_in_place. Large blobs and parallel Git operations can put memory and blocking thread capacity under pressure. Introduce bounded read/write APIs, streaming where useful, and one service-level IO budget.
Cleanup queues Pending source blob and repo storage deletions live as JSON on the metadata lock row. Cleanup state shares contention with unrelated metadata work and is hard to retry or inspect. Move cleanup into typed job rows with idempotency keys, attempts, next run time, and failure diagnostics.

North star

The target architecture should preserve the thing Scope already does well: domain behavior owns the rules. The change is that persistence stores durable facts and read models serve repeated queries.

Domain behavior Owns invariants, transitions, access decisions, projection semantics, and required effects.
Canonical facts Repo record, settings, commits, file changes, visibility events, blobs, tokens, members, invites, staged updates.
Read models Current file trees, history summaries, file diffs, preview metrics, access summaries, prepared Git refs.
Workers Import, publish, read-model rebuild, Git artifact generation, source blob cleanup, repo storage cleanup. Run these from a dedicated Railway worker service rather than the request-serving API.
Thin delivery HTTP, CLI, Git smart HTTP, and web routes translate inputs, call domain operations, persist facts, and render results.

Canonical write model

  • Keep source graph and policy canonical, but store them as rows instead of one large JSON document.
  • Require every mutation to produce a repo revision and explicit effects.
  • Commit facts and outbox jobs in the same transaction.
  • Rehydrate a full aggregate only where domain behavior needs the whole aggregate to make a transition.

Disposable read model

  • Treat projection trees, history pages, file diffs, and Git artifacts as rebuildable products of canonical facts.
  • Key read models by repo id, repo revision, source (live or review), and audience class.
  • Return explicit rebuilding or unavailable states when the read model is stale or missing.
  • Version read-model semantics so old caches can be deleted destructively.

Roadmap

I would treat the Linear project as a sequence of small destructive changes. Each step should leave the app working, but we should not preserve old storage paths for compatibility.

Phase 0

ADA-33 - Benchmark hot paths

Build a repeatable simulated benchmark for auth, repo list, repo detail, files, projection preview, history, clone credential, info/refs, upload-pack, receive-pack, publish, and cleanup. Complete: local benchmark harness is merged and available through ./dev/scope-dev bench.

Baseline is available for external latency, response bytes, status mix, Git smart HTTP, git subprocess, and first-push receive-pack timing.
Phase 1

ADA-34 - Remove auth writes from normal reads

Active now: resolve established Clerk users and valid CLI sessions through read-only lookups. Move user snapshot refresh, expired-session cleanup, and last-used writes out of the normal request path. Complete: established Clerk and CLI auth reads no longer perform normal request-path snapshot refreshes, expired-session cleanup, or last-used writes. Explicit lifecycle writes remain named and tested.

Success means normal authenticated GETs do not take the metadata write lock just to know who the principal is. This was the smallest cut that made the global-lock problem easier to see.
Phase 2

ADA-35 - Replace the global metadata lock

Active now: audit every metadata lock caller, move uniqueness and referential integrity into constraints where possible, lock individual repo rows for repo mutations, and move cleanup state toward typed rows.

Success means unrelated repos can read and mutate without queuing behind a catalog-wide lock row. Do not split repository JSON in this phase unless it is necessary to remove the lock safely.
Phase 3

ADA-36 - Normalize repository persistence

Split the large repository JSON aggregate into durable fact tables. Keep mutation APIs behavior-owned: SQL stores facts, but domain modules still decide transitions and side effects.

Use destructive pre-alpha reset. Delete obsolete JSON columns instead of shadow-writing old and new shapes.
Phase 4

ADA-37 - Materialize read models

Precompute current file trees, projected commits, history summaries, file diff indexes, and visibility summaries by repo revision and audience class.

Normal UI reads should become indexed reads. Rebuild code should prove read models are disposable.
Phase 5

ADA-38 - Add outbox and workers

Commit side-effect jobs with metadata changes. Workers handle import processing, read-model rebuild, Git artifact generation, and cleanup with explicit states and retries. The implementation target is a separate scope-worker Railway service, not an in-process API background loop.

This is where pushes and publishes stop tying up request handlers for all expensive work. This phase includes the repo refactor needed for shared API and worker code, plus CI and Railway deployment for the worker service.
Phase 6

ADA-39 - Bound Git and object-store IO

Add service-level concurrency budgets for receive-pack, upload-pack, projection builds, raw snapshot restore, bundle generation, and object-store operations.

The user-visible behavior should be predictable backpressure rather than random request stalls.

Phase 2 work order

The first Phase 1 PR should be narrow: preserve user-facing auth behavior, but stop turning every known authenticated read into a metadata write transaction. Do not start the global lock replacement or repository JSON split in this phase. The first Phase 2 PR should be an audit-backed lock reduction, not a repository schema rewrite. Preserve behavior, prove which invariants need serialization, then replace catalog-wide locking with narrower database-enforced guarantees.

1

Map lock callers

Trace optional_scope_user, require_scope_user, and require_clerk_scope_user through repo list/detail/session/projection routes. Trace every acquire_metadata_read_lock and acquire_metadata_write_lock caller. Classify each as repo-scoped mutation, global invariant, cleanup state, reset/admin, or auth lifecycle.

2

Move invariants into SQL

Add read-only lookup paths for known Clerk identities and CLI sessions. Use unique indexes, foreign keys, row filters, and row locks for invariants already represented as rows: users, auth identities, CLI sessions, members, invites, repo identity, and token rows.

3

Replace global cleanup state

Run the Phase 0 harness before and after. The important comparison is authenticated read routes: repo list, settings, staged update, owner projection preview, and session endpoints. Move pending source-blob and repo-storage deletion state out of the metadata lock JSON and into typed cleanup/job rows with idempotency keys, attempts, next-run timing, and diagnostics.

Implementation rule: a valid existing user/session should not mutate scope_users, scope_auth_identities, scope_cli_sessions, or scope_metadata_locks during normal read authorization. no Phase 2 change should preserve the global lock as a convenience boundary. If a caller still needs catalog-wide serialization after the audit, name the invariant explicitly and prove why a constraint, scoped row lock, or typed job row is not enough.

Guardrails

These are the architectural rules I would hold while doing the data work.

Keep

  • Domain modules own rules, allowed transitions, visibility semantics, and required side effects.
  • Effects are explicit values before they become jobs, object deletes, notifications, or artifact builds.
  • Read models are rebuildable from canonical facts and versioned by semantics.
  • Tests target behavior and failure modes: permissions, visibility leaks, stale reads, idempotent retries, and cleanup safety.

Reject

  • Backward compatibility layers for obsolete pre-alpha storage shapes.
  • SQL branches that independently invent business rules already owned by domain code.
  • Read-model tables that become a second canonical source of truth.
  • Inline long-running Git, S3, or cleanup work when a transaction can enqueue explicit work instead.

Conversation agenda

The first-pass defaults are accepted. The next conversation should focus on ADA-34 execution details and what counts as a normal read versus an explicit auth lifecycle event. The Phase 2 conversation should focus on what the metadata lock actually protects today, which invariants can be moved into SQL immediately, and which cleanup/job tables should exist before worker extraction.

  1. Translate the accepted role vocabulary into table names, permission columns, and read-model audience names. Classify metadata lock callers into global invariant, repo-scoped invariant, cleanup/job state, and admin/reset categories.
  2. Define the first two audience classes explicitly: public-projection and private-reader-projection. Decide which lock removals are pure SQL constraint/transaction changes and which require a small domain persistence refactor.
  3. Agree that the source graph stays canonical and read models are disposable.
  4. Use the merged Phase 0 harness as the before/after proof for ADA-34. Use tests and code inspection as primary proof for ADA-35; the benchmark harness remains a regression signal, not the only success measure.
  5. The first cut is chosen: remove normal auth request writes before global lock or repo JSON work. The next cut is chosen: shrink the metadata lock before normalizing repository JSON.
  6. Define the scope-worker service boundary, shared Rust modules, Railway variables, and GitHub Actions deployment flow before implementing ADA-38. Define cleanup/job rows now so the later scope-worker service has a durable queue instead of inheriting JSON state from the lock row.

How I will start Phase 2

Phase 1 should remove avoidable writes from established authenticated reads without weakening auth behavior. Phase 2 should remove catalog-wide serialization without weakening repository, cleanup, or auth lifecycle invariants. The work starts with the lock callers, not with a broad repository table redesign.

Start 1

Inventory metadata lock callers

Run ./dev/scope-dev bench and record the authenticated read cases. Then inspect api/src/auth/scope.rs and api/src/db/auth.rs to list every route that reaches resolve_clerk_user or verify_cli_session_token. Trace every acquire_metadata_read_lock and acquire_metadata_write_lock caller in api/src/db. Record the invariant each caller is protecting and whether the operation is global, repo-scoped, auth-scoped, cleanup-scoped, or admin-only.

Output: lock caller matrix with owner module, route surface, protected invariant, and proposed replacement mechanism.
Start 2

Delete lock uses covered by constraints

Introduce a read-only lookup for existing Clerk identities that loads scope_auth_identities and scope_users without updating the user snapshot. For already-normalized tables, replace lock-dependent uniqueness and existence checks with database constraints and transactional error handling. Start with users, auth identities, CLI sessions, members, invites, and repo identity rows.

Output: fewer write-lock callers without changing user-facing behavior or repository JSON shape.
Start 3

Scope repository mutation locks

Change active CLI session verification to read the session and user without writing last_used_at_unix during every request. For operations that still mutate the repository aggregate, lock the specific repository row or related rows instead of the metadata lock row. Keep the full domain transition intact; only the transaction boundary should narrow.

Output: unrelated repositories stop blocking each other on one global update lock during normal repo mutations.
Start 4

Move cleanup state into typed rows

Add targeted tests for established Clerk users, first Clerk login, verified-email conflict, valid CLI session use, expired CLI session handling, and revoke. Replace pending_repo_storage_deletions and pending_source_blob_deletions JSON on scope_metadata_locks with typed cleanup rows. Include idempotency key, attempts, next run time, last error, and completion state.

Output: cleanup becomes inspectable and retryable without making the lock row a queue.
Start 5

Prove lock removal behavior

Run ./dev/scope-dev bench again and compare the authenticated read cases against the Phase 0 baseline. Add tests around duplicate repo/user/member/invite races where practical, cleanup retry safety, and repo mutation isolation. Run the Phase 0 harness as a regression check, but treat lock-caller deletion and transactional tests as the primary Phase 2 proof.

Output: ADA-35 closes with fewer lock callers, explicit invariants, and no user-facing behavior drift.
Guardrail

Do not overbuild Phase 2

Do not introduce the worker service, job tables, read models, or global lock replacement inside ADA-34. Do not normalize the full repository source graph, materialize read models, or build the Railway worker service inside ADA-35. Typed cleanup/job rows are allowed only because they replace lock-row queue state and prepare the later worker phase.

Output: a destructive but narrow lock-boundary refactor. If a lock removal is risky, keep that caller explicit and move to the next lower-risk caller instead of broadening into Phase 3.
