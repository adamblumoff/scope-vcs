Executive read

Scope has a strong domain center, but the persistence and read side still behave like a serialized catalog with large repository aggregates. The next architecture push should keep domain behavior authoritative while breaking storage, reads, and side effects into smaller durable facts.

My recommendation: start the Data Architecture project now, but do not jump straight into a large table rewrite. First benchmark the hot paths, then remove request-path auth writes and the global lock pressure, then split repository persistence and materialize read models.

Pre-alpha advantage: this is the right time to do destructive schema replacement. Keeping compatibility for the current JSON-heavy repository row would preserve the exact shape we need to escape.

Current state

Recent work has cleaned up several boundaries: object-store modules are split, staged push rules moved into domain ownership, and collaboration HTTP glue was condensed. The remaining data problem is less about file size and more about the shape of state and side effects.

Strong

Domain model

StoredRepository, SourceGraph, visibility policy, staged updates, collaboration rules, and projections hold the important product rules in durable domain code.

Mixed

Persistence

Postgres has real tables for users, auth identities, members, invites, CLI auth, and repositories. The repository table still stores graph, policy, tokens, pending import, snapshot, and staged update as JSON.

Costly

Reads

Repo detail, files, projection preview, history, and Git clone paths commonly rehydrate the full repository aggregate and replay projection logic on demand.

Hot

Side effects

Git subprocesses, object-store reads and writes, projection Git repo creation, bundle restore, and cleanup can run inline with request handling.

Typical hot path today

This is the pattern I would expect to dominate current latency and capacity. It is not wrong for pre-alpha, but it is the shape that will cap growth.

Auth
Clerk JWT is verified, then resolve_clerk_user can enter a metadata write transaction to create or update the Scope user snapshot.
Metadata
Repo reads take the metadata lock row in share mode. Mutations and auth flows take it in update mode. Many unrelated repos therefore still coordinate through one row.
Aggregate
A repository row is loaded and decoded into StoredRepository, including graph, policy, token state, snapshots, pending import, and staged updates.
Compute
Projection, file tree, preview, and history are computed from the source graph on request. Public and owner views often repeat the same work.
Git and blobs
Git clone paths may build or restore bare repos using subprocesses and object-store reads. Object-store calls currently return whole byte buffers.

Pressure points

These are the places where the architecture is likely to bend first as data or concurrency grows.

Area Current behavior Why it matters Likely fix
Global metadata lock One scope_metadata_locks row protects broad metadata invariants. Unrelated users and repos can queue behind auth writes, repo mutations, cleanup updates, and large reads. Use constraints, row-level locks, scoped repo locks, and typed job tables instead of one catalog lock.
Large repository JSON scope_repositories stores graph, policy, visibility events, tokens, snapshots, pending import, and staged update as JSON. Normal detail/list/settings reads become sensitive to repo history size and JSON decode cost. Split canonical facts into domain-owned tables and rehydrate full aggregate only for mutation paths.
Derived reads Projections, file trees, history, and clone artifacts are built from canonical graph data on demand. Read cost tracks total repo history instead of result size. Materialize read models keyed by repo revision, source, and audience class.
Inline Git work Git subprocesses and bundle/projection repo creation happen during request handling. A burst of pushes or clones can block normal UI traffic. Add outbox jobs, workers, prepared artifacts, concurrency limits, and explicit backpressure responses.
Object-store interface ObjectStore::get returns Vec<u8>; S3 uses blocking reqwest inside block_in_place. Large blobs and parallel Git operations can put memory and blocking thread capacity under pressure. Introduce bounded read/write APIs, streaming where useful, and one service-level IO budget.
Cleanup queues Pending source blob and repo storage deletions live as JSON on the metadata lock row. Cleanup state shares contention with unrelated metadata work and is hard to retry or inspect. Move cleanup into typed job rows with idempotency keys, attempts, next run time, and failure diagnostics.

North star

The target architecture should preserve the thing Scope already does well: domain behavior owns the rules. The change is that persistence stores durable facts and read models serve repeated queries.

Domain behavior Owns invariants, transitions, access decisions, projection semantics, and required effects.
Canonical facts Repo record, settings, commits, file changes, visibility events, blobs, tokens, members, invites, staged updates.
Read models Current file trees, history summaries, file diffs, preview metrics, access summaries, prepared Git refs.
Workers Import, publish, read-model rebuild, Git artifact generation, source blob cleanup, repo storage cleanup. Run these from a dedicated Railway worker service rather than the request-serving API.
Thin delivery HTTP, CLI, Git smart HTTP, and web routes translate inputs, call domain operations, persist facts, and render results.

Canonical write model

  • Keep source graph and policy canonical, but store them as rows instead of one large JSON document.
  • Require every mutation to produce a repo revision and explicit effects.
  • Commit facts and outbox jobs in the same transaction.
  • Rehydrate a full aggregate only where domain behavior needs the whole aggregate to make a transition.

Disposable read model

  • Treat projection trees, history pages, file diffs, and Git artifacts as rebuildable products of canonical facts.
  • Key read models by repo id, repo revision, source (live or review), and audience class.
  • Return explicit rebuilding or unavailable states when the read model is stale or missing.
  • Version read-model semantics so old caches can be deleted destructively.

Table direction

This is not a final schema. It is the table vocabulary I think we should use when we start ADA-36 and ADA-37. The important part is ownership: each table should either be canonical domain fact, disposable read model, or explicit side-effect work.

Group Candidate tables Purpose
Repository identity repo_records, repo_settings, repo_revisions Small rows for list/detail/settings routes. Revision becomes the join point between canonical changes, read models, and job idempotency.
Access repo_members, repo_invites, repo_access_snapshots Members and invites are already partly normalized. Access snapshots can keep UI and Git authorization from needing the full source graph.
Auth and tokens auth_identities, cli_sessions, repo_push_tokens, repo_clone_tokens Move repo token state out of repository JSON. Normal auth verification should be a narrow indexed lookup, not a catalog write.
Source graph logical_commits, logical_commit_parents, file_changes, source_blobs Store the canonical source graph in queryable rows. Preserve domain ownership by writing through domain transition outputs.
Visibility visibility_rules, visibility_events, path_visibility_epochs Make current visibility and historical visibility inspectable. Epochs can support fast public history and audit semantics.
Review state pending_imports, pending_import_files, staged_updates, staged_file_changes Keep review surfaces explicit instead of stuffing pending work into the repo row.
Read models projection_files, projection_commits, history_entries, file_diff_index Serve UI and API reads in O(page size or result size), not O(total repo history).
Git artifacts git_snapshots, git_projection_artifacts, git_artifact_refs Track prepared bundles, bare repo cache keys, snapshot blobs, and artifact readiness.
Work queues outbox_jobs, cleanup_jobs, job_attempts Make side effects explicit, idempotent, retryable, and visible to operators. The worker service owns job execution; API requests own job enqueueing inside the metadata transaction.
Worker service deployment api crate shared modules, worker binary, service-specific Railpack config, and GitHub Actions deploy workflow. Make scope-worker deploy like scope-api, scope-cli, and scope-web: CI first, Railway deploy second, no manual service drift.

Roadmap

I would treat the Linear project as a sequence of small destructive changes. Each step should leave the app working, but we should not preserve old storage paths for compatibility.

Phase 0

ADA-33 - Benchmark hot paths

Build a repeatable simulated benchmark for auth, repo list, repo detail, files, projection preview, history, clone credential, info/refs, upload-pack, receive-pack, publish, and cleanup.

Output: baseline latency, lock wait time, object-store bytes, Git subprocess count, memory pressure, and a small fixture matrix.
Phase 1

ADA-34 - Remove auth writes from normal reads

Resolve Clerk users from indexed identity rows without updating user snapshots on every request. Move snapshot refresh and session last-used writes to explicit events or sampled updates.

This is the lowest-risk way to reduce write contention quickly.
Phase 2

ADA-35 - Replace the global metadata lock

Move uniqueness and referential integrity into constraints, lock individual repo rows for repo mutations, and use typed job rows for cleanup state.

Success means unrelated repos can read and mutate without queuing behind a catalog-wide lock row.
Phase 3

ADA-36 - Normalize repository persistence

Split the large repository JSON aggregate into durable fact tables. Keep mutation APIs behavior-owned: SQL stores facts, but domain modules still decide transitions and side effects.

Use destructive pre-alpha reset. Delete obsolete JSON columns instead of shadow-writing old and new shapes.
Phase 4

ADA-37 - Materialize read models

Precompute current file trees, projected commits, history summaries, file diff indexes, and visibility summaries by repo revision and audience class.

Normal UI reads should become indexed reads. Rebuild code should prove read models are disposable.
Phase 5

ADA-38 - Add outbox and workers

Commit side-effect jobs with metadata changes. Workers handle import processing, read-model rebuild, Git artifact generation, and cleanup with explicit states and retries. The implementation target is a separate scope-worker Railway service, not an in-process API background loop.

This is where pushes and publishes stop tying up request handlers for all expensive work. This phase includes the repo refactor needed for shared API and worker code, plus CI and Railway deployment for the worker service.
Phase 6

ADA-39 - Bound Git and object-store IO

Add service-level concurrency budgets for receive-pack, upload-pack, projection builds, raw snapshot restore, bundle generation, and object-store operations.

The user-visible behavior should be predictable backpressure rather than random request stalls.

What I would do first

The first PR should not be a schema replacement. It should make the bottlenecks measurable and create a shared vocabulary for the rest of the work.

1

Benchmark fixture

Generate repos with a small, medium, and large source graph: different file counts, commit counts, visibility mixes, pending review state, and member access.

2

Hot path runner

Exercise auth, repo list/detail, projection preview, history, files, Git clone, push, publish, and cleanup under light and burst concurrency.

3

Report format

Emit a stable JSON or Markdown report with latency, query count, lock wait, decoded bytes, object-store bytes, Git subprocesses, and memory high-water notes.

Useful threshold: if a normal read route deserializes source graph JSON or reconstructs a projection tree, the benchmark should make that visible. We do not need perfect production load testing yet; we need enough measurement to stop arguing blind.

Guardrails

These are the architectural rules I would hold while doing the data work.

Keep

  • Domain modules own rules, allowed transitions, visibility semantics, and required side effects.
  • Effects are explicit values before they become jobs, object deletes, notifications, or artifact builds.
  • Read models are rebuildable from canonical facts and versioned by semantics.
  • Tests target behavior and failure modes: permissions, visibility leaks, stale reads, idempotent retries, and cleanup safety.

Reject

  • Backward compatibility layers for obsolete pre-alpha storage shapes.
  • SQL branches that independently invent business rules already owned by domain code.
  • Read-model tables that become a second canonical source of truth.
  • Inline long-running Git, S3, or cleanup work when a transaction can enqueue explicit work instead.

Conversation agenda

The first-pass defaults are accepted, so the next conversation should focus on execution details instead of reopening those direction choices.

  1. Confirm the product role model: public, member/maintainer, owner, and which actions each can take. Translate the accepted role vocabulary into table names, permission columns, and read-model audience names.
  2. Decide whether read models are audience-class based or user-specific. Define the first two audience classes explicitly: public-projection and private-reader-projection.
  3. Agree that the source graph stays canonical and read models are disposable.
  4. Define the first benchmark fixture and the report numbers we care about.
  5. Pick the first destructive schema cut: auth writes, global lock, or repo JSON columns.
  6. Decide how aggressive we want to be about workers in the first pass. Define the scope-worker service boundary, shared Rust modules, Railway variables, and GitHub Actions deployment flow before implementing ADA-38.

How I will start Phase 0

Phase 0 should produce trustworthy baseline numbers and a fixture we can reuse after every architecture cut. It should not change behavior or schema.

Start 1

Inventory the hot paths

Map the exact handlers and functions for auth resolution, repo list/detail, files, projection preview, history, Git info/refs, upload-pack, receive-pack, publish/apply, and cleanup drain.

Output: a short route/function matrix tied to the benchmark cases.
Start 2

Build deterministic fixtures outside source code

Create small, medium, and large repo fixtures with generated commits, files, visibility changes, pending review state, and member access. Keep benchmark/test code outside production source unless there is a specific justification.

Output: repeatable fixture setup that can target Postgres, with a memory fallback only for cheap smoke coverage.
Start 3

Add a benchmark runner

Run each hot path under light and burst concurrency, measure elapsed time, status, response bytes, Git subprocess count, object-store bytes, and observable lock/transaction wait where Postgres exposes it cheaply.

Output: JSON plus a readable Markdown summary under a generated output directory, not committed benchmark results.
Start 4

Keep CI sane

Do not make full benchmarks part of normal CI. Add a fast compile/smoke path so fixture and runner code does not rot, then run the full benchmark manually before and after each data-architecture phase.

Output: normal checks stay fast; architecture PRs still have before/after evidence.
Start 5

Use the baseline to choose the first cut

If auth writes dominate, start ADA-34. If lock wait dominates across unrelated repos, start ADA-35. If JSON decode and projection rebuild dominate normal reads, start ADA-36/37 sequencing with the narrowest destructive split.

Output: the Phase 1 choice is evidence-driven instead of based on whichever concern feels loudest.
