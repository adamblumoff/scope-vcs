Locked Decisions

The product model is repository-centric. Repositories are owned by one Scope user. Collaboration is granted through explicit repository membership. Public access is computed from file visibility and publication state, not stored as a role.

Ownership Repos are user-owned for now. The owner is derived from Repository.owner_user_id.
Membership Collaborators are RepositoryMember records. Every member can read private files.
Permissions Member powers are individual booleans, not role names or a rank-ordered enum.
Public Public means no repo membership. Public readers see only the published public projection.

Explicitly Rejected

  • No workspace or organization ownership in this pass.
  • Collapse PendingFirstPush and PendingPublish into one unpublished repository state.
  • No Public, Reader, Writer, Maintainer, or Owner role ladder.
  • No persisted owner membership row.
  • No member-management powers delegated to members.
  • No compatibility with the current pre-alpha schema.
  • No historical member table; removed members are hard-deleted.

Current State And Failure Mode

The current implementation has useful pieces, but the ownership and role model are duplicated. That creates drift risk and has already shaped inconsistent checks.

Current source What it says Problem
RepoRecord.owner_user_id Canonical repo owner user id. Some code derives owner access from this even if membership rows drift.
RepoMembership.role = Owner Also says who owns the repo. Owner-only mutations can reject the owner if this row is missing or stale.
Policy.owner_id Gives one principal file-policy bypass behavior. File policy knows about ownership even though ownership is a repo concern.
RepoRole enum Reader | Writer | Maintainer | Owner. Maintainer has no distinct behavior, and role rank hides product decisions.
Core cleanup: ownership must have one source of truth, and file visibility policy must stop deciding who owns a repo. Policy should answer visibility; repository access should answer authority.

Target Data Model

The durable domain model should make access explicit and small: repository ownership, repository membership, member permission grants, invite state, and file visibility policy.

Invariants

  • A repository has exactly one owner user id.
  • The owner is not stored as a repository member.
  • A repository member always has private read access.
  • Members receive no mutation permissions by default.
  • Member permissions are independent booleans.
  • Public access is the absence of ownership or membership.
  • File visibility policy stores only public/private file visibility, not ownership.
  • Owners cannot invite members until the repository is published.
  • Owner-only settings remain owner-only, even if a member has every member permission.

Computed Capabilities

Routes, Git auth, review actions, and UI state should consume one computed access object instead of checking raw rows or booleans in place. That keeps the rules singular.

Access Shape

RepositoryAccess
  actor: Owner | Member | Public
  can_read_private_files: bool
  can_push: bool
  can_change_file_visibility: bool
  can_apply_changes: bool
  can_update_repo_settings: bool
  can_manage_members: bool
  can_delete_repo: bool
Actor Private read Push Visibility Apply/reject/publish Settings Members Delete
Owner Yes Yes Yes Yes Yes Yes Yes
Member Yes Permission Permission Permission No No No
Public/nonmember No No No No No No No

Important Permission Interactions

  • can_push does not imply can_apply_changes.
  • can_apply_changes does not imply can_change_file_visibility.
  • A member with push but without apply permission must not bypass review through auto-apply settings.
  • Changing member permissions bumps change_version and publishes a repo-change event.
  • Git tokens identify a user; they do not embed permissions. Authorization checks current membership and permissions at request time.

Invite Flow

Invites are email-based. The invite link finds the pending invite, but verified email proves the invitee. Pending invites grant no repository access.

Repository Invite

RepositoryInvite

  repo_id: RepositoryId
  invited_email: String
  invited_email_normalized: String
  invited_by_user_id: ScopeUserId
  permissions: RepositoryMemberPermissions
  state: Pending | Accepted | Revoked | Expired
  token_hash: String
  created_at_unix: u64
  updated_at_unix: u64
  expires_at_unix: u64
  accepted_by_user_id: Option<ScopeUserId>
  accepted_at_unix: Option<u64>
  revoked_at_unix: Option<u64>

Default Invite Permissions

RepositoryMemberPermissions
  can_push = false
  can_change_file_visibility = false
  can_apply_changes = false

Invite Rules

  • Only the repository owner can create, refresh, or revoke invites.
  • Invites can only be created after the repository is published.
  • A pending invite is unique by repo_id and normalized email.
  • Re-inviting the same pending email refreshes the invite, token, expiry, and permissions.
  • Inviting the owner email is rejected.
  • Inviting an email that already belongs to a repository member is rejected.
  • Accepted, revoked, and expired invites cannot be accepted.
  • Acceptance requires a signed-in Scope user with a verified matching email.
  • Acceptance creates a RepositoryMember with the invite permissions.
  • If the invitee is not a Scope user yet, sign-up/sign-in resumes the invite flow afterward.
Owner-facing UI copy: membership always includes private file access. The invite and member settings screens should say this next to the permission toggles, because all toggles off still grants private read access.

Repository Lifecycle And Collaboration

Collaboration must be wired through repository lifecycle without inventing a second access model. The target lifecycle is only Unpublished and Published; whether an unpublished repo has not received a first push yet or has a pending first import is represented by pending import/token data.

Lifecycle state Owner behavior Member behavior Public behavior
Unpublished Can initialize, review first import, publish, and delete. No members yet; invites are blocked until published. No read.
Published All capabilities. Private read plus explicit mutation permissions. Public projection only.

Push And Review Rules

  • First import review and publication are owner-only because members cannot be invited before publish.
  • Owner pushes follow repository settings.
  • Member push requires can_push.
  • If a member lacks can_apply_changes, their push must stage for review even when the repo is configured to auto-apply owner pushes.
  • Applying or rejecting a staged update requires owner access or member can_apply_changes.
  • Changing staged or live file visibility requires owner access or member can_change_file_visibility.

Domain Boundaries To Create

The heavy lift is not just schema. The code needs one domain access boundary that every outer layer calls. HTTP, Git, SSE, response shaping, and web UI should not invent their own rules.

Boundary Owns Deletes/replaces
Repository access domain module Owner/member/public resolution and capability computation. Scattered role rank checks such as role >= Writer.
File visibility policy module Effective public/private visibility for paths. Policy.owner_id and owner bypass behavior inside policy.
Repository member domain module Adding, removing, and updating members and permissions. Generic RepoMembership role records.
Repository invite domain module Email invite state machine, acceptance, revocation, expiry. Invitation JSON stored inside repository rows.
Capability-shaped API responses UI action availability without leaking raw internals. UI checks against role strings such as repo.role === "Owner".

Implementation Phases

This should be done as a planned destructive refactor, not as a local patch. Each phase should leave the repo compiling and test failures should point at the next intentional behavior gap.

01

Domain Model Rewrite

Replace RepoRole and RepoMembership with RepositoryMember and RepositoryMemberPermissions. Remove owner rows and move ownership decisions to Repository.owner_user_id.

  • Update domain structs in api/src/domain/store.rs.
  • Collapse RepoPublicationState to Unpublished | Published.
  • Split file policy from repo ownership in api/src/domain/policy.rs.
  • Add capability computation tests before wiring HTTP.
02

Persistence Schema Reset

Destructively replace membership persistence and make invites first-class rows. Do not migrate old role data.

  • Replace stored PendingFirstPush/PendingPublish values with the new unpublished shape by resetting schema.
  • Create scope_repository_members with permission columns or a typed JSON permissions column.
  • Create scope_repository_invites with token hash, normalized email, state, timestamps, and acceptance fields.
  • Remove repository-row invitations JSON.
  • Update schema inventory, entity models, load/save paths, and test support.
03

Authorization Boundary

Add a single access service that returns RepositoryAccess. Convert state helpers, repo actions, Git auth, SSE, and response builders to consume it.

  • Replace ensure_repo_owner with capability-specific guards.
  • Replace can_write_path with explicit push and visibility capabilities.
  • Keep public/nonmember principal behavior, but rename concepts around audience/access.
04

Git, Review, And Events

Rewire Git read/write and review routes so token-based access checks current membership and permission grants at request time.

  • Clone token read access requires owner or current member.
  • Push requires owner or member can_push.
  • Apply/reject/publish requires owner or member can_apply_changes.
  • Visibility changes require owner or member can_change_file_visibility.
  • Member permission changes bump version and publish repo-change events.
05

Invite And Member APIs

Add owner-only endpoints for invite creation, refresh, revocation, member removal, and member permission updates. Add invite lookup/acceptance endpoints for signed-in users.

  • Normalize emails consistently.
  • Reject invite creation while the repository is unpublished.
  • Hash invite tokens and return plaintext tokens only when creating or refreshing.
  • Reject owner email and existing member email invites.
  • Hard-delete removed members and revoke or invalidate repo-scoped tokens.
06

Web Surface

Build owner-only collaboration settings without presenting private read access as optional. Refresh action availability immediately when repo-change events report capability changes.

  • Owner settings: published-only members list, invite form, permission toggles, remove member.
  • Member settings: read-only section showing private read access and each enabled/disabled permission.
  • Invite page: token lookup, sign-in/sign-up resume, verified email acceptance.
  • Repo pages: use API capabilities instead of role string checks.
  • Copy: "Members can always view private files in this repository."
07

Cleanup And Contract Regeneration

Remove old role names and compatibility branches, regenerate TypeScript API types, and run static deletion checks.

  • No production references to Reader, Writer, Maintainer, or RepoRole.
  • No owner check that depends on a member row.
  • No file policy owner id.
  • No invitation JSON stored on repository rows.

Verification Plan

Tests need to prove the invariants, not just exercise endpoints. The important failures are privilege escalation, stale token permissions, stale UI actions, and accidental public/private leaks.

Domain Persistence HTTP Git Web Static checks
  • Owner access is derived from owner_user_id only.
  • Repository lifecycle exposes only Unpublished and Published.
  • Repository members always read private files.
  • All member permission defaults are false.
  • Public/nonmember sees only public published files.
  • Pending invite grants no private access.
  • Invite acceptance requires verified matching email.
  • Duplicate pending invite refreshes instead of inserting another pending row.
  • Existing member invite is rejected.
  • Owner email invite is rejected.
  • Owner cannot invite members while the repo is unpublished.
  • Removed member immediately loses private read.
  • Removed member cannot use old clone token.
  • Permission downgrade refreshes UI capabilities.
  • Member without push cannot push over Git HTTP.
  • Member with push can push, but cannot auto-apply without apply permission.
  • Member without visibility permission cannot change live or staged visibility.
  • Member without apply permission cannot publish, apply, or reject.
  • Member cannot update repo settings.
  • Member settings show a read-only permission summary.
  • Member cannot invite, revoke, remove, or edit member permissions.
  • Owner-only delete remains owner-only.
  • Public event streams stay redacted.
  • Removed member event stream closes or loses permissioned action state.
  • Generated TypeScript API types match Rust response/request types.
  • Static search finds no old role ladder names in production code.

Checks Before Coding

These are the remaining implementation-time decisions to confirm before changing product code. They are narrow enough to resolve in the implementation plan.

Check Default recommendation Why it matters
Unpublished collaboration Owner-only. Members cannot be invited until the repo is published. Prevents first-import review from needing member permissions before collaboration exists.
Member table permission storage shape Dedicated boolean columns. Columns make queries, indexes, and drift checks clearer than opaque JSON.
Invite expiry duration 7 days. Needs a concrete default for token lifecycle tests and UI copy.
Token invalidation on member removal Delete repo-scoped clone tokens for removed user. Defense in depth even though auth checks current membership.
Capability response naming Return capabilities, not role. Keeps web from reconstructing domain rules from labels.
This plan intentionally treats the current app as pre-alpha: destructive schema replacement is preferred over compatibility shims, legacy role translation, or partial dual-read behavior.
