Binary file round-tripping

Make a PNG survive push -> projection -> clone byte-for-byte by treating source blobs as bytes everywhere and making text diff a review-only view.

Goal Demo-safe binary support without compatibility scaffolding

Simple Summary

The object store already handles raw bytes. The system fails binary files because import, projection, review diffs, and one dead JSON projection endpoint still pretend every blob is UTF-8 text with a line count.

Push Accept Git blob bytes exactly as Git provides them. Validate path, mode, object count, and byte limits only.
Store Persist blob metadata as object key, SHA-256, Git OID, mode, and byte length. No line count.
Project Rebuild Git projection repos by writing raw bytes into Git object storage, not by converting blobs to strings.
Review Render text diffs only when both sides are text. Show a binary diff placeholder otherwise.

Conversation Context

The earlier prioritization put binary round-tripping immediately after the catalog read path and schema canonicalization because it is the last table-stakes demo gap. The concrete demo invariant was: a repo containing a PNG must survive push, projection, and clone byte-for-byte. Until this works, every demo can fail on an ordinary file.

Working assumption: this is still pre-alpha with no valuable data. Shape changes should be destructive. If a text metadata column or compatibility reset exists only to support the old text-only model, remove it.

Revision note: the plan now explicitly tracks two watch-items: response-time diff cost on public history routes, and clear push-time errors for binaries that exceed the object-store byte cap.

Repo Findings

Import

api/src/git/import/repo_io.rs already reads blobs as Vec<u8> from git cat-file --batch, but parse_git_cat_file_batch rejects each blob if std::str::from_utf8 fails. The test pushed_tree_rejects_non_utf8_blobs_before_pending_import locks in that bad behavior.

Projection

api/src/git/upload.rs builds projection repos by converting every SourceBlob through source_blob_text, then passing UTF-8 bytes to git hash-object --stdin. That breaks public and member Git clone for binary files even if import is fixed.

Domain

SourceBlob stores line_count, and StagedFileChange stores line_diff. Both are stale text-only presentation metadata. Git integrity needs sha256, git_oid, git_file_mode, and size_bytes. Review counts should be computed from bytes when the API is building a review response.

API

Review and history file-diff responses return Option<String> content. The unused /v1/repos/{owner}/{repo}/projections route also serializes projection content as strings and is not included in the generated TypeScript API contract.

Storage

crates/scope-core/src/object_store/source_blobs.rs is already byte-safe at the object-store boundary. The useful helper is source_blob_bytes; the dangerous helper is source_blob_text because it makes text decoding look like a core blob operation.

Recommended Direction

Use one rule: source blobs are bytes. Text is a derived presentation mode for review only.

Why this is simple

Git already gives us blob OIDs and byte lengths. The object store already stores bytes. The projection cache already uses git hash-object --stdin. We only need to stop converting blob bytes to strings before those boundaries.

Why this deletes code

Once line counts and staged line diffs leave the domain, the fallback diff path, DB columns, cleanup-row mappings, staged update callback plumbing, reset compatibility exceptions, and UTF-8 rejection tests all go away.

What stays

Path validation, Git file mode validation, blob byte limits, SHA-256 verification, Git OID calculation, visibility policy, projection history, and Git clone credentials stay intact.

What changes visibly

Text files continue to render text diffs. Binary files show a clear "binary file not rendered" state with size and OID metadata, while Git clone returns the real bytes.

Watch Items

These are not blockers for pre-alpha, but they should shape the implementation so the next move is obvious instead of painful.

Diff cost

Removing stored line counts means review and history responses compute text diffs from object-store bytes on demand. A page with N changed files can now fetch up to 2N blobs, each using the object-store permit budget and each doing decode work.

That is acceptable for owner review pages at dogfooding scale, but history diffs are public and already rebuild per request. Write the response helper as a pure function over old and new byte slices, with fetching kept outside it. Then diff content can be lifted into a materialized read model later without changing the text/binary rules again.

Large files

Binary files are how users will hit SCOPE_OBJECT_STORE_MAX_BYTES. Keep the existing byte limit, but make the push-time error name the path and the configured limit before object-store insertion turns it into a generic oversized-object failure.

Sequencing

This changes SourceBlob across the domain, DB row models, cleanup jobs, and schema contract. Do not land this in parallel with catalog/per-repo hydration work; whichever branch goes first should merge cleanly before the other rewrites the same persistence boundary.

Implementation Plan

1. Tests first

Add a failing real-Git test that writes fixed PNG-like bytes, pushes them over HTTP, publishes, clones through the Git read path, and asserts byte equality. Add a second published-update test that modifies the binary file and verifies the projected clone sees the new bytes.

  • Replace the current non-UTF-8 rejection test with acceptance.
  • Assert private binary files stay out of public projections.
  • Assert executable text mode coverage still passes after byte changes.
  • Add an almost-text file with valid UTF-8, CRLF line endings, and trailing whitespace to prove byte passthrough does not normalize content.
2. Blob model

Remove line_count from SourceBlob, remove line_diff from StagedFileChange, and delete the generic LineDiff callback from staged-update domain code. Keep only durable blob and review facts in core.

  • Update repo_object_for_bytes to stop counting lines.
  • Update repository snapshot and source-blob cleanup table specs.
  • Simplify stage_staged_update so it no longer takes a diff callback.
  • Add exact-column schema drift so deleted columns trigger a pre-alpha reset.
3. Import

Delete the UTF-8 validation at the end of parse_git_cat_file_batch. The parser should only verify Git batch headers, object kind, OID, byte size, content delimiter, and trailing data.

Keep current rollback cleanup behavior. The failure cases that remain are real validation failures: unsupported paths, Git links, unsupported modes, oversized blobs, too many files, or Git command failures.

For oversized blobs, reject during import validation with a message that names the file path and byte limit. Do not let the first visible error be a generic object-store cap failure.

4. Projection

Change projection cache construction to carry bytes, not strings: 

ProjectionTreeFile {
  bytes: Vec<u8>,
  git_file_mode: String,
}

Populate bytes with source_blob_bytes and feed those bytes into git hash-object -w --stdin. Bump PROJECTION_CACHE_SEMANTICS_VERSION so old text-built caches are ignored.

5. Review API

Replace raw string diff content with an explicit content union: 

ReviewFileContentResponse =
  | { kind: "text", text: string }
  | { kind: "binary", oid: string, size_bytes: number }

Text decoding and line-diff calculation happen in the response layer. If UTF-8 decoding succeeds and the blob is under the existing exact-diff byte limit, return text and compute counts. Otherwise return binary metadata and count the file as changed without inventing line additions or deletions. History file diffs and review file diffs should share the same helper.

Keep the helper pure over byte slices and metadata. The route code can fetch blobs, but the diff classifier should not depend on HTTP state, DB state, or object-store handles.

6. Review UI

Update ReviewFileDiffDrawer to call @pierre/diffs only when both old and new content are text or empty. If either side is binary, render a compact binary state with the path, change kind, size, and abbreviated OID. Keep the file tree and visibility controls unchanged.

7. Remove dead route

Delete get_projection, ProjectionResponse, ProjectedCommitResponse, and ProjectedChangeResponse. Remove the router entry and the test that asserts /projections returns raw text. The product already uses typed file lists, preview summaries, commit history, and Git clone for real projection access.

API And UI Shape

The API should be explicit about whether content is renderable text. Do not base64 binary content into JSON; that would make a dead preview path expensive and would not help Git clone correctness.

Verification

The minimum proof is byte equality through real Git, not just API metadata. The tests should exercise both first push and published update paths.

Manual demo
  1. Create a repo with image.png and README.md.
  2. Push it through the first-push HTTP remote.
  3. Publish the repo.
  4. Clone with the public or scoped token path.
  5. Compare sha256sum image.png before and after clone.

Not Doing

  • No MIME detection. Git does not need MIME type for round-tripping.
  • No image preview UI. The demo needs clone correctness first.
  • No base64 binary content in JSON projection responses.
  • No materialized diff read model yet. The pure helper keeps that option easy later.
  • No backwards-compatible migration for line_count; this is pre-alpha.
  • No branch support, mirror support, or cache version checks beyond bumping projection semantics.
