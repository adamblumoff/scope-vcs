Review draft

Request collaboration should feel like joining a shared branch.

This pass improves the branch-backed Request workflow so developers can start, join, pull, push, sync with main, submit, check status, and delete Requests without learning server refs or hidden lifecycle machinery.

The product should expose one mental model: a Request is a visible website object with a backing branch. start creates the object and local branch, join lets another developer work on it, pull gets website request updates, push publishes local request updates, sync-main catches up with main, and submit asks maintainers for review.

Prepared 2026-07-08

Builds on branch-backed Requests phases 1-6

Primary surfaces CLI, Request API, Request web UI

Main goal simple multi-developer local collaboration

Non-goal browser-based Git editing or conflict resolution

Why This Pass Exists

The current Request implementation has the right foundation: durable request records, server-owned request refs, clean maintainer merge, and CLI-owned branch writes. The rough edge is that the user has to infer too much. Request creation, branch creation, website visibility, collaboration, and submission are separate concepts, but the commands do not yet make those boundaries obvious.

Add

Source-specific branch commands

Center the workflow on start, join, pull, push, sync-main, submit, status, and delete. Request branch sync and main sync should be separate commands.

Modify

start opens a shared workspace

Change scope request start from local-only branch creation into the command that creates a visible working Request and prints the URL.

Delete

Remove hidden Git/ref UX

Delete or replace CLI/API assumptions that a request does not exist until submit. Users should not need to understand reserved refs, request refs, or finalize steps.

Target Mental Model

A Request has two separate gates. First, a developer starts a working Request so other developers can see and join it. Later, the Request is submitted for maintainer attention. Only that second step charges public stake.

1

Start

Create website Request, backing branch, and local branch from the auth-appropriate main projection.

2

Link

The website URL and join command are available immediately. The Request can be discussed before maintainer review.

3

Join

Another developer fetches the request branch locally and gets clear permission messaging.

4

Pull / Push

Developers exchange committed request branch changes through the website Request.

5

Sync Main

The request branch can be rebased onto the latest allowed main projection when main moves.

6

Submit

Public author stakes credits once and asks maintainers to review. Owners and members skip the credit economy.

Proposed state split: replace the current user-facing Reserved idea with a real Working state. Working means visible on the website, backed by a request ref, not yet in maintainer-review queue, and no public stake charged.

CLI Command Shape

The CLI should lead every command with identity, repo access, base projection, local branch, and allowed next actions. Users should never have to infer whether they are working against public main, private main, or the website Request branch. The command names should identify which source of truth they touch.

scope request start --title "..."

Creates a server Request in Working, publishes the initial backing branch, creates the local branch, stores metadata, and prints the website URL.

scope request join req_123

Fetches the backing branch, creates or updates a local branch, stores request metadata, and tells the user whether they can edit.

scope request pull

Gets the latest Request branch from Scope/the website. Fast-forward when possible; if local unpublished commits exist, rebase them on top of the website Request branch or stop with conflict instructions.

scope request push

Publishes committed local changes to the website Request branch when the user has edit access. If the website Request moved first, reject and tell the user to run scope request pull.

scope request sync-main

Rebases the local request branch onto the latest allowed main projection. It does not publish request changes; run scope request push after resolving and committing.

scope request submit

Moves Working to Submitted, charges public stake once, and places the Request in the maintainer queue.

scope request status

Shows the Request URL, state, edit permission, local branch, whether local work is ahead or behind the server request branch, whether main moved, and the next recommended command.

scope request delete

Deletes an unsubmitted working Request or withdraws a submitted Request through the same canonical API used by the web UI.

scope request share

Convenience command that prints the Request URL and exact join command. It should also be included in status output.

API And Domain Model

Collaboration needs domain support, not only CLI wrapping. The product language should be simple: some people can edit a Request and some people can only view it. Maintainers can comment, pull, and edit by role. Public users can pull the branch, edit, and comment only when they are the author or an invited editor.

Recommended

Working Requests plus role-aware edit access

Add a Working request state and a small edit-access relation for public editors. Maintainers can pull, push, and comment by role. Public authors and invited public editors can pull, push, and comment; other public users cannot.

Boundary

Economics stay author-owned

Edit access changes who can work on the branch; it does not split stake or settlement. The Request author owns the credit outcome for this pass.

Request state Add Working. Remove or hide Reserved from user-facing flows. A started Request needs to exist on the website before first submitted review.
Submission command Change submit from create-and-finalize to working-to-submitted. Stake should be charged once when maintainer attention is requested, not when the developer creates a working branch.

Implementation Phases For This Pass

This is designed as one cohesive pass, but the order matters. The domain and API changes should land before CLI polish so the CLI does not encode rules the domain does not own.

Phase A: Domain foundation

Core
A.1

Introduce Working as the started-but-not-submitted Request state and make it the result of request start.

A.2

Split start/finalize semantics: start creates the Request and request ref; submit transitions to maintainer review and charges stake.

A.3

Add request edit-access permissions: maintainers can pull/edit/comment by role; public users can pull/edit/comment only when they are the author or an invited editor.

A.4

Add delete/withdraw domain mutation with tests for Working, Submitted, NeedsResponse, Resolved, and already Withdrawn.

Phase B: Canonical API

API
B.1

Replace reserve/finalize user flow with start/submit endpoints while keeping internal ref admission explicit.

B.2

Add request delete endpoint used by both CLI and web.

B.3

Add read fields needed for collaboration: local join command, viewer edit permission summary, current request head, and state.

B.4

Add the smallest edit-access mutation needed for two-person public collaboration. The UI should expose it as adding or removing public editors, not as a broad collaboration subsystem. Maintainers do not need invitations.

Phase C: CLI collaboration UX

CLI
C.1

Modify scope request start to create the website Request, publish the initial backing branch, and print the URL.

C.2

Add join and delete, then make pull and push the normal request collaboration loop for fetching and publishing committed request work.

C.3

Rename the existing main-projection rebase behavior from sync to sync-main so request-origin pull and main-origin sync are impossible to confuse.

C.4

Improve status to show local ahead/behind request branch, whether main moved, permission level, URL, and next recommended command. It should recommend exactly one of pull, push, sync-main, or submit.

Phase D: Web request surface

Web
D.1

Show Working Requests differently from Submitted Requests in list and detail views.

D.2

Add Work Locally command block and simple Can edit messaging to request detail. Public users without editor access should not see a usable branch pull command.

D.3

Add delete/withdraw action with explicit confirmation and reuse the canonical API.

Phase E: Verification and cleanup

Closeout
E.1

Add CLI integration tests for start, join, pull, push, sync-main, submit, status, delete, and stale branch conflict handling.

E.2

Add API/domain tests for lifecycle, edit permissions, delete/withdraw, and stake timing.

E.3

Remove obsolete reserved/finalize CLI wording, old update-only tests, and any unreachable API client methods.

Deletion Track

Resolved Product Decisions

These decisions are settled for this pass and should be treated as implementation constraints.

Decision 1

Use sync-main

The main-update command is scope request sync-main. Request-origin movement remains pull and push.

Decision 2

pull is safe but helpful

Fast-forward when possible, rebase unpublished local commits onto the website Request when safe, and stop with conflict instructions when not safe. Never publish during pull.

Decision 3

Who can edit a Request?

Maintainers can comment, pull, and edit by role. Public users can pull the branch, edit, and comment only when they are the author or an invited editor.

Decision 4

Add editors from the web first

Public editor invitations should be web-first, with a small CLI helper only if needed. The recipient's primary command remains scope request join req_123.

Decision 5

Delete working, withdraw submitted

Working Requests are hard-deleted, including their backing branch. Submitted Requests are withdrawn and keep timeline history.

The main tradeoff is command count versus ambiguity. This plan keeps one extra command because it makes the two origins explicit: pull/push talk to the website Request, and sync-main talks to main.
