Destructive implementation plan

Replace web review and UI visibility edits with config-owned CLI review.

Scope should stop storing pending reviews for the web app. The config file becomes the only file-visibility write surface, the CLI becomes the review surface, and the server becomes the validator and atomic applier.

This plan intentionally deletes old surfaces instead of preserving compatibility. There are no users and no valuable data, so the architecture should move to the model we actually want.

Core replacement: delete server-side review queues and web publish controls. Replace them with a reviewed push intent generated by scope push.

Non-negotiable outcomes

These are the target rules the implementation should optimize around.

Config owns visibility

File and folder visibility changes come only from .scope/repo.json. The web UI can display policy, but it cannot change it.

CLI owns review

Review happens before upload in scope push. The user reviews content changes, config changes, and destructive rewrite requests locally.

History is forward-only by default

Old commits keep the visibility they had when created. Destructive public-history cleanup requires an explicit config entry.

The new push flow

The important change is that the server no longer persists a pending review. The server applies a reviewed push or rejects it.

1

Read local state

CLI reads committed .scope/repo.json, current HEAD, and the Scope base snapshot.

2

Compute plan

CLI builds changed files, config policy diff, forward visibility boundaries, and requested destructive rewrites.

3

Review locally

TUI shows what will be public, private, forward-only, and destructively rewritten. Flags can skip prompts.

4

Create push intent

CLI sends the reviewed manifest to Scope. Scope returns a one-time Git credential bound to that exact manifest.

5

Upload Git data

CLI performs Git receive-pack with the intent credential. The server verifies HEAD, config digest, base, and manifest.

6

Apply atomically

Server writes graph, visibility boundaries, config, snapshot, and optional destructive rewrites in one mutation.

Replacement domain shape

The durable domain should own the config schema, validation, visibility timeline, and destructive rewrite commands. Outer layers only parse inputs, call the domain, persist results, and render output.

New domain-owned concepts

  • RepoConfig: parsed config with kind, version, visibility rules, and explicit history rewrite requests.
  • ReservedScopePath: domain rule that /.scope/repo.json is always private and cannot be overridden.
  • ReviewedPush: content changes plus reviewed config effects plus expected base/head/config digest.
  • VisibilityBoundary: forward-only public/private boundary emitted when config changes affect a live path.
  • HistoryRewriteRequest: explicit destructive path-scoped operation such as redact-public-history.

The existing FileChange.visibility should become the historical source of truth for that commit. Public projection should stop asking the current policy whether an old public change is still readable. 

{
  "$schema": "https://scope.dev/schemas/repo-config.v1.json",
  "kind": "scope.repo-config",
  "version": 1,
  "visibility": {
    "default": "private",
    "rules": [
      { "path": "/README.md", "visibility": "public" },
      { "path": "/src/**", "visibility": "public" },
      { "path": "/notes/**", "visibility": "private" }
    ]
  },
  "history": {
    "rewrites": [
      {
        "path": "/leaked-secrets/**",
        "action": "redact-public-history"
      }
    ]
  }
}

Implementation phases

This order deletes broad surfaces early, then replaces the core path with a single reviewed push command.

Introduce config parsing and validation in the domain

Add RepoConfig, reserved path validation, glob/path matching, schema version handling, and config digesting in crates/scope-core. Reject config attempts to expose /.scope/repo.json.

Switch projection to commit-forward visibility

Change public projection so historical public file changes stay visible based on their stored FileChange.visibility. Use visibility boundaries for future tree changes. Keep private to public as a baseline; public to private as a forward public deletion.

Add reviewed push command

Replace stage_staged_update, apply_staged_update, and publish_import with one domain command that accepts a reviewed push manifest and either applies it completely or fails without mutating the repo.

Replace receive-pack persistence with push intents

Add GET /push-context and POST /push-intents. The intent returns a one-time Git credential bound to expected base, head, config digest, and reviewed policy effects. Receive-pack applies only if those facts match.

Rewrite scope push

CLI loads push context, reads committed config from HEAD, computes local diff, runs TUI review, creates the push intent, then performs Git push with the intent credential. Add flags for noninteractive approval and browser-free operation.

Make scope init config-first

Stop printing a web review URL. Create the repo and remote, add or validate .scope/repo.json, and direct the user to commit the config and run scope push. First reviewed push publishes the repo automatically.

Delete web review and visibility mutation surfaces

Remove review route, review actions, review API client, publish/apply/reject buttons, live visibility toggles, settings visibility toggles, and review redirects. Replace repo detail with a read-only file/projection view.

Destructively simplify schema and generated types

Drop pending review columns and settings fields. Regenerate TypeScript API contracts after removing review DTOs. Update schema contract tests to assert the smaller shape.

Replace tests instead of patching them

Delete review publish/diff tests that assert old behavior. Replace with domain tests for config validation, reserved path privacy, commit-forward projection, destructive rewrites, push intent validation, and CLI push flows.

Tradeoffs we are choosing

This approach is simpler and more coherent, but it deliberately gives up some flexibility.

Tradeoff What we gain What we give up
CLI-only review Local, fast, config-aware review with no server reconciliation queue. No browser-based review, no pending review link, no review handoff across devices.
Config-only visibility changes Versioned, auditable policy changes that travel with the repo. No quick web toggle for one-off visibility changes.
No server-side staged update Smaller server state and fewer stuck states to clean up. A rejected review cannot be resumed later because it never reaches the server.
Forward-only history by default Policy changes no longer surprise users by rewriting old history. Previously public leaks remain public until an explicit destructive rewrite is requested.
Explicit destructive rewrites Dangerous changes are visible, path-scoped, and reviewable. Rewrite implementation is more complex and can invalidate projection/cache identities.
Reserved config file is always private Public readers cannot inspect policy or destructive rewrite intent. Public clones are intentionally not a byte-for-byte view of the source tree.
Push intent credential Server can bind Git upload to reviewed local decisions. Push flow gets one more API round trip and a new failure mode when intent facts drift.
Collapse review settings One push path, fewer settings, fewer branches in domain logic. No per-repo choice between auto-apply and review-before-apply.
Potentially collapse visibility permission into push permission Simpler collaboration model: if you can push config, you can affect policy. Less granular safety for collaborators. If this feels wrong, keep a server-side config-change permission without adding UI toggles for file paths.
Destructive schema cleanup No compatibility layer and less dead model weight. Existing dev data and tests around old review behavior are thrown away.

Verification plan

The new tests should prove the new model directly instead of trying to preserve old review behavior.

Domain tests

  • Config parser accepts only known schema/kind/version.
  • Reserved config path is always private.
  • Public to private creates forward public deletion only.
  • Private to public creates public baseline only.
  • Destructive rewrite removes named public history only.

API/Git tests

  • Push intent rejects stale base/head/config digest.
  • Receive-pack without a valid intent cannot mutate repo.
  • First reviewed push publishes repo.
  • Published reviewed push applies atomically.
  • Public clone never contains .scope/repo.json.

CLI/Web tests

  • scope push fails when config is missing or uncommitted.
  • TUI review lists visibility and destructive rewrite effects.
  • Flags can run push noninteractively.
  • Web has no review route and no visibility controls.
  • Settings has no review/default-visibility toggles.
