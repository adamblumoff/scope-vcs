Simple Summary

Scope has already reduced schema duplication. The remaining read path problem is that normal repo pages still ask persistence for StoredRepository, which loads graph, policy, pending review, staged update, settings, tokens, members, invites, and blob metadata even when the route only needs summary, settings, or files.

Recommended Direction

Add narrow read APIs in MetadataStore for repo summary, repo list, repo settings, and live file lists. These APIs should use plain Postgres transactions and normal MVCC snapshots, not the global metadata lock row. Keep full aggregate hydration for mutation, review, history, Git, and publish/apply paths until each path has a specific reason to move.

Current Context

The canonical schema contract work landed. schema_contract/* now owns table identity and DDL generation. That means this plan is not another broad schema cleanup. It is a read-unit cleanup.

Target Shape

The control decision is to stop coordinating ordinary reads through one catalog-sized unit. Postgres already gives consistent read snapshots. Normal repo reads should use those snapshots and select only the rows needed for the response.

Commands and mutations

  • Hydrate StoredRepository.
  • Run domain transitions and invariants.
  • Write canonical facts and enqueue required effects.
  • Keep using scoped write coordination until each write path is narrowed deliberately.

Queries and UI reads

  • Return read-shaped structs, not StoredRepository.
  • Use plain transactions and row snapshots.
  • Read repo rows, settings, members, and projection read-model rows directly.
  • Do not decode source graph JSON for list/detail/settings/files.
MetadataStore
  repo_summary(owner, name, viewer_user_id)       -> Option<RepoSummaryRead>
  repo_summaries_for_user(user_id)                -> Vec<RepoSummaryRead>
  repo_settings(owner, name, viewer_user_id)      -> RepoSettingsRead
  repo_live_files(owner, name, viewer_user_id)    -> Vec<ProjectionViewFile>

Full hydration remains available as:
  repository(owner, name) -> Option<StoredRepository>
  mutate_repository(...) -> domain-owned mutation result

Implementation Scope

This should be one small foundation slice. The goal is not to redesign every read path. The goal is to remove the broad aggregate from the routes users hit constantly.

Route or API Change Full aggregate after this?
GET /v1/repos Replace repositories_for_user() with a summary query over repository rows, owner rows, member rows, settings, and review-state booleans. No
GET /v1/repos/{owner}/{repo} Add a single-repo summary read. Compute access from owner and member rows. Read public visibility using policy plus projection read-model availability where needed. No
GET /v1/repos/{owner}/{repo}/settings Read repo identity, access facts, default visibility, and settings row. Do not load graph or review JSON. No
GET /v1/repos/{owner}/{repo}/files Use the existing live projection read model by repo version and audience class. If the read model is missing or stale, fall back explicitly to full hydration for correctness. Only fallback
MetadataStore::repository() Keep it for paths that still need domain graph data, but remove the global read lock from the Postgres implementation. A transaction snapshot is enough for read-only hydration. Yes, by design

Keep the memory metadata implementation in parity by deriving the same read structs from AppCatalog. That preserves fast tests and local memory dev while the Postgres path becomes narrow.

Implementation Steps

Step 1

Create read-shaped core structs

Add a focused module such as crates/scope-core/src/db/repo_reads.rs. Define small structs for repo identity, access, summary, settings, and file-list read context. These are not API response structs; they are persistence read outputs that the API can render.

  • RepoSummaryRead: id, owner handle, name, lifecycle, default visibility, change version, access, review booleans.
  • RepoSettingsRead: default visibility plus settings fields.
  • RepoReadAccess: the same access decisions the domain currently exposes, computed from owner/member rows.
  • RepoFileReadContext: repo id, version, audience class, and access needed for projection file reads.
Step 2

Implement narrow Postgres reads without the metadata lock

Add read methods on MetadataStore. Each Postgres method should start a transaction only to get a consistent snapshot across multiple queries. Do not call acquire_metadata_read_lock.

  • Use repository rows for identity and lifecycle fields.
  • Use member rows for access and list membership.
  • Use settings rows for settings responses.
  • Use pending_import IS NOT NULL and staged_update IS NOT NULL style checks instead of decoding those JSON blobs.
  • Use existing projection read-model rows for files where available.
Step 3

Keep memory metadata behavior aligned

Implement the same read methods for the memory store by deriving read structs from AppCatalog. This keeps the local dev and test story intact without forcing Postgres-shaped tests everywhere.

Step 4

Move the normal repo routes first

Convert list_repos, get_repo, get_settings, and get_files to the narrow APIs. The API layer should translate read structs into existing response structs. Public API JSON should not change.

Step 5

Leave full hydration where it is still honest

Review routes, history routes, Git upload-pack/receive-pack, projection preview, clone credentials, publish/apply, and visibility mutations can keep using StoredRepository. Those paths currently need graph, policy, pending review, staged update, token, or mutation semantics. Move them only after a specific bottleneck or read-model need appears.

Step 6

Remove old broad read entry points

After route conversion, delete repositories_for_user() returning StoredRepository. Remove the global read lock from read-only full hydration. Keep mutate_repository and other command APIs on the write-coordinated path.

Source Blobs Later

The first canonical schema slice after this should probably be source_blobs, but it should not be hidden inside this read-foundation work. Binary support made blob metadata more important, and today most SourceBlob values still live embedded in repo JSON. That deserves a separate, explicit plan.

Future Slice Reason Do Not Do Now
scope_source_blobs Canonical blob metadata would connect directly to binary correctness, cleanup safety, and object-store reference tracking. Do not normalize commits, file changes, or visibility events as part of this read slice.

Verification

This should prove that behavior is unchanged while the read path is narrower. The tests should focus on route outputs, access decisions, and absence of broad hydration from converted routes.

  • cargo fmt --all -- --check
  • cargo check --workspace
  • cargo test -p api repo_lifecycle repo_visibility repo_collaboration
  • cargo test -p api review_publish review_diff git_receive git_http as regression coverage for untouched full-hydration paths.
  • cargo test --workspace --all-targets before opening a PR.
  • pnpm -C web check because generated API types and route data assumptions may move.
  • ./dev/scope-dev bench to confirm repo list/detail/files/settings still pass the local hot-path harness.

Add focused tests that assert over the public behavior, not the implementation. The useful implementation guard is a local audit: converted routes should no longer call find_repo() or MetadataStore::repository().

Not Doing

  • No logical_commits, file_changes, or full source graph normalization.
  • No new cache layer or version-checked in-memory repo cache.
  • No branch support, mirror support, org/team work, or GitHub mirror work.
  • No API response shape changes for repo list/detail/settings/files.
  • No removal of write-path metadata coordination in this slice.
  • No attempt to move review, history, Git, publish/apply, or collaboration mutation paths off full hydration.
  • No compatibility shim for old read APIs once their callers are gone.
