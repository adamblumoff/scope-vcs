01

Outcome and boundaries

A single visual system should make Scope feel like one deliberate product rather than a set of route-specific utility pages.

Outcome. Repository identity and primary navigation live in one persistent top bar. Repository facts occupy a compact metadata strip. Each route renders a border-led workbench beneath it, with typography and interaction—not cards or chromatic decoration—creating hierarchy.
Domain boundary. Visibility, permissions, request states, mergeability, settlement, invitations, deletion, clone availability, and live refresh remain owned by existing API/domain code. UI modules translate those facts; they do not infer new rules.
Theme boundary. Dark mode is canonical because it is the approved direction. Light mode remains supported with a derived neutral palette and identical component geometry. The existing theme toggle moves into the global user menu.
Resolved presentation behavior. README files render with react-markdown and remark-gfm; raw HTML remains disabled and links pass through an explicit safe-URL policy. Search is omitted from the shipped top bar. A reusable search-control component may be built and visually verified for future use, but it remains unmounted and has no product affordance until real search behavior exists.
Resolved workspace-tab behavior. Code files and History diffs use one shared tab-strip interaction. Selecting a file opens it once and activates it; selecting an open file reactivates its existing tab. The active item remains URL-owned for reload and deep linking, while the wider open-tab collection is ephemeral repository-scoped session state. History tab collections are additionally scoped by audience and commit so diffs from different projections never mix.
Resolved top-bar craft. The top bar is one layered, machined surface rather than a row of outlined boxes. Repository identity becomes an unboxed breadcrumb with subdued owner and emphasized repository. Route navigation becomes a compact raised rail with a dimensional active item. Keep its geometry stable across Code, Requests, and History by placing route-specific controls in the route workbench header. Clone becomes a dark satin split control aligned with the Repository title instead of a disconnected light rectangle in the persistent shell. Load the selected variable interface and mono fonts locally—never merely name an unavailable font—and use restrained 120–160ms surface/icon transitions.

03

Design principles

These rules apply across routes and are more important than copying any isolated pixel from the mock.

Content is the surface

Use rows, panes, type, and rules. Avoid wrapping every section in a card. Repository files, diffs, requests, members, and sessions should remain visibly connected.

Neutral interaction

Platinum communicates selection, focus, and primary action through luminance. Red, green, warning, and info colors remain sparse and semantic.

Craft over decoration

Distinctiveness comes from spacing, icon consistency, layered one-pixel borders, typography, and complete states—not glows, gradients, or ornamental panels.

One navigation model

Global identity, repository context, and primary sections stay in the top system on every repository route. Never introduce side navigation.

Dense, not cramped

Use 44–48px rows, readable 14–16px copy, consistent baselines, and 24–32px section rhythm. Preserve scan speed without shrinking important text.

States are first-class

Every component specifies default, hover, focus, active, selected, disabled, pending, success, empty, and error behavior before route migration.

Typography

  • Bundle and actually load a neutral variable interface face such as Geist Sans or Inter Tight plus a matching mono face; do not rely on an unavailable Inter declaration falling back to Segoe UI.
  • Fonts must be local build assets with font-display: swap; no runtime request to a third-party font service.
  • Use mono only for paths, branch names, hashes, code, shortcut keys, and machine identifiers.
  • Brand: 17–18px. Repository identity/body: 14–15px. Route navigation: 12.5–13px. Metadata: 11–12px. Page titles: 28–36px.
  • Prefer variable medium and semibold weights; reserve bold for critical labels.

Geometry

  • 4px base spacing; primary rhythm at 8, 12, 16, 24, 32, and 48px.
  • Controls: 36px compact, 40px normal, 44px coarse pointer.
  • Radii: 4px tags/keycaps, 6–7px controls, 9px dialogs. No decorative 16–24px rounding.
  • Shadows are limited to menus, dialogs, and primary floating layers.

07

Route-by-route overhaul

Each route adopts the same shell and primitives while preserving its own behavior and information architecture.

Repositories / home

Global top bar with Repositories active. Replace rounded repository cards with a dense resource list: repository identity, visibility, actor access, request count, lifecycle exceptions, and contextual primary action. Empty state presents CLI installation as one guided sequence.

Repository / Code

Exact approved workbench. File browser left; open-file tab strip and active viewer right. Clicking a tree file opens it once or reactivates its existing tab. Render safe README Markdown; code remains mono and scrollable; binary/unavailable states preserve the surrounding browser. The root README is the initial tab when available. Clone is the right-aligned Repository workbench-header action so switching routes never shifts the persistent top bar.

Requests list

Requests becomes a resource list with stable columns at desktop and labeled stacked metadata on mobile. Status and mergeability use compact semantic indicators. Preserve title, request name/ID, audience, role, OID, update time, and stake.

Request review

Keep Overview / Changes / Activity as a local tab row below the request header. Overview uses section rows for facts and workflow actions. Changes adopts the file-browser/diff workbench. Activity becomes a clean chronological rail without card bubbles.

History

Use a commit list + commit files + tabbed diff viewer workbench. Selecting a changed file opens one diff tab keyed by audience + projected commit ID + path; selecting it again activates the existing tab. Tabs never mix files from different commits or audiences. Closing the active tab activates the nearest remaining tab and updates path; closing the last clears path. Audience selection belongs in the context strip or workbench header. Mobile moves through commit → files → horizontally scrollable tabs → diff as explicit route state while preserving URL search state.

Settings and collaboration

Settings remains a top-level repository tab only for allowed actors. Access, invite, member, pending invite, and danger sections use border-led rows. Permission switches retain domain values; destructive actions remain visually isolated.

Account and CLI sessions

Global top bar with account context. One-time login and sessions become two workbench sections. Session rows prioritize label and recency; revoke remains a semantic destructive icon with confirmation.

CLI authorization

Use the standard shell for browser-initiated flows and a focused single-column workbench. Preserve all callback handoff states, timers, fallback link, sign-in gate, code entry, and terminal return messaging.

Invitations

Public-compatible top bar and a focused invitation document. Show repository identity, target email, permissions, authentication requirement, acceptance action, pending, and failure states without a promotional card.

Authentication

Replace the large gradient marketing split with a quiet, balanced identity surface that matches the product. Style Clerk through scopeClerkAppearance; do not fight Clerk’s interaction behavior or inject a fake sidebar.

Errors and unavailable routes

Route errors use the same top shell when repository context is known. Preserve recovery links and object identity. Errors use a red icon/rule, not a fully red panel.

Dialogs, menus, toasts

Use one elevated surface language: strong border, restrained shadow, 7–9px radius, clear heading, concise copy, and aligned footer. Toast color is semantic only; routine success remains neutral with a small green marker.

09

Accessibility and interaction contract

The visual overhaul is incomplete unless keyboard, screen-reader, reduced-motion, and coarse-pointer behavior are designed alongside it.

  • Retain skip link, banner, navigation, main, section, list/tree, dialog, and status semantics.
  • Active route and selected resource use shape/position in addition to color.
  • All interactive controls have visible platinum focus rings at 3:1 against adjacent surfaces.
  • Text reaches WCAG AA; small muted text is audited separately in both themes.
  • Coarse-pointer targets are at least 44×44px without inflating desktop density.
  • File trees support arrow navigation, Enter/Space selection, and correct expanded state.
  • Scrollable tab rows expose labels and do not trap horizontal scrolling.
  • Workspace tabs use tablist/tab semantics, roving focus, Left/Right/Home/End navigation, and a separately named close button. Close remains available on focus and coarse pointers even when hover-only chrome is visually reduced on desktop.
  • Closing an active workspace tab moves focus and selection predictably to the nearest remaining tab; closing the last returns focus to the file navigator and clears the active URL path.
  • Async errors remain adjacent to the action and use live-region announcements.
  • Pending buttons preserve width and expose loading text, not spinner-only status.
  • Reduced motion disables pane transitions, hover translation, and loading animation beyond essential progress.
  • Destructive dialogs name the target and restore focus to the initiating control.
  • Mobile stacked workbenches provide an explicit, keyboard-accessible way back to the resource list.

Implementation sequence

Each phase is independently reviewable and verified in the running app. Do not attempt a single blind CSS rewrite.

P0

Baseline and fixtures

Open the running local app, capture the current authenticated/public routes, and write a manual QA checklist before visual changes.

  • Desktop 1440×1000, tablet 900×900, mobile 390×844, narrow 320×720.
  • Public and authenticated repository access; populated and empty data.
  • Record all current route/URL behavior that must survive.
P1

Tokens and primitives

Replace palette, load real local variable fonts, refine typography, radii, focus, motion, and shared controls. Build Storybook-free local primitive test route only if needed, then remove it.

  • Verify computed font families and eliminate silent system fallback.
  • Validate dark and light contrast.
  • Component states first; no route migration yet.
P2

Shell migration

Build the layered top-bar surface, unboxed repository breadcrumb, raised route-nav rail, route-invariant top-bar geometry, satin Clone split control in the Repository workbench header, repository context strip, responsive navigation row, account menu, and global skip/main structure.

  • Migrate home and repository shell first.
  • Verify public visitor, member, owner, and unauthenticated variants.
  • Inspect all hover/focus/active transitions at reduced and normal motion settings.
P3

Workbench foundation

Build split workbench, resource list, file browser, shared workspace tabs, file viewer, status, feedback, and action primitives.

  • Preserve active file/diff selection in URL state and keep open-tab order in scoped session state.
  • Test activation, duplicate prevention, close-neighbor selection, last-tab clearing, overflow, keyboard focus, storage reconciliation, and mobile/coarse-pointer behavior.
  • Add safe README Markdown/GFM rendering with raw HTML disabled.
  • Build and visually verify the unmounted future SearchControl.
  • Verify binary, missing, error, empty, and large text files.
P4

Repository workflows

Migrate Code, Requests list/detail, Changes/diffs, History, Settings, members, and invite management.

  • Adopt the same workspace-tab primitive in Code and History; History identity must include audience + commit + path.
  • Split request detail and member modules by behavior while touching them.
  • Verify every permission-gated action before moving on.
P5

Global workflows

Migrate repositories home, account/CLI sessions, CLI login, invitations, auth, errors, loading, toasts, dialogs, and empty states.

  • Validate Clerk’s hosted components and popovers in both themes.
  • Confirm callback handoff timing is unchanged.
P6

Subtraction and polish

Delete legacy shell/primitives, remove unused tokens and classes, audit file sizes, and resolve visual inconsistency.

  • No compatibility wrappers.
  • No dead alternate theme or route-specific style islands.
P7

Review and closeout

Run functional checks, then manually inspect and exercise every route in the running local development app across the target viewports. Keep fixing visual, responsive, keyboard, and interaction defects until the manual QA checklist is clean. Run autoreview with a 15-minute timeout while manual verification continues.

Acceptance criteria

The overhaul is complete only when the whole product—not just the Code screenshot—meets the approved standard.

  • Every route uses the new top-navigation system; no legacy two-row app/repository shell remains.
  • The top bar reads as one crafted, route-invariant surface: repository identity is unboxed, route navigation has dimensional active treatment, Clone is visually integrated with the Repository workbench header, and no silent font fallback flattens the typography.
  • No route introduces a persistent sidebar or dashboard-card layout.
  • Platinum/steel owns interaction; semantic red and green never indicate ordinary selection.
  • Code, requests, history, and diffs use one coherent workbench and resource-row language.
  • Code and History share one polished workspace-tab interaction: no duplicate tabs, active URL state survives reload/navigation, open order survives the session, close behavior is deterministic, and History tabs are isolated by audience + commit.
  • Settings, account, auth, invites, dialogs, errors, loading, and empty states feel like the same product.
  • Public, owner, and member views expose exactly the actions allowed by existing domain permissions.
  • URL-driven file/view/audience/commit selections still survive reload and browser navigation.
  • No horizontal page overflow at 320px; coarse-pointer controls meet 44px targets.
  • Dark and light themes pass contrast review and preserve identical hierarchy.
  • No inert search, fake metadata, speculative action, compatibility layer, or dead style system ships.
  • All automated behavior checks pass and the completed manual local-dev QA checklist has no known console errors, page errors, overflow, clipped content, broken states, or unresolved visual defects.
  • The running application matches the approved mock’s restraint, hierarchy, and component quality.
