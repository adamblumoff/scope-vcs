Persistence schema surface reduction

Executive Read

The next architecture move should be subtraction. Scope does not need another data-architecture phase before dogfooding. It needs the persistence layer to stop making every schema change cross four parallel representations.

Decision

Treat ADA-45 as a one-track maintainability project: collapse schema representation divergence in scope-core without changing product behavior.

Why now

Binary file support, read-path cleanup, and eventual repo graph normalization will all touch persistence. Doing those while table shape is repeated across broad files compounds the cost.

What not to do

Do not turn this into Phase 7, a migration framework, or full commit/file-change normalization. Make future changes cheaper first.

Current Friction

The problem is not that the DB module has many concepts. The problem is that the same table and column shape is repeated in places that have to stay manually synchronized.

Representation Current owner Change tax
Raw schema builders schema.rs, schema_repository_facts.rs, schema_read_models.rs, schema_outbox.rs Every table and column is declared in migration-builder code.
Schema inventory and drift checks metadata_schema.rs Table names, column names, and drift expectations are duplicated in a second broad file.
SeaORM entities entities.rs The persisted row shape is repeated as entity structs and field names.
Domain structs and mapping glue domain/store.rs, repository_rows.rs, entity conversion impls Behavior-owned structs are correct, but conversion code makes schema changes feel larger than the product change.

Concrete pressure

entities.rs, metadata_schema.rs, and schema.rs are all near the project line cap. They are still under 1000 lines, but they already behave like broad catch-all files.

System pressure

The six-phase data-architecture program cleaned up real issues, but continuing that style now would add process complexity for a product whose current load is one developer dogfooding.

Approaches

There are three reasonable ways to attack the representation divergence. The recommended path is intentionally less ambitious than a schema generator and more aggressive than file shuffling.

Recommended

Entity-centered schema ownership

Make SeaORM entity modules the primary place a table's row shape is described. Keep schema creation helpers close to each entity family, and use shared helpers for drift/reset checks instead of maintaining a second table inventory by hand.

  • Best balance of low risk and real deletion.
  • Preserves domain structs as behavior-owned types.
  • Works with destructive pre-alpha reset semantics.
Fallback

Family split without ownership change

Split entities.rs, schema.rs, and metadata_schema.rs by table family while keeping the same representation count.

  • Lower risk.
  • Improves file size and navigation.
  • Does not solve the root duplication.
Reject for now

Custom schema code generation

Build a mini DSL or generator that emits schema builders, inventory, entities, and mapping shells.

  • Could remove duplication in theory.
  • Adds a private toolchain and more machinery.
  • Wrong level of investment before dogfooding.

Recommended Work Order

Keep this as a small sequence of behavior-preserving PRs. Each PR should reduce the schema edit surface or make the next deletion easier.

1

Inventory the duplicated shape

Add a short internal audit in the PR description, not a new long-lived phase doc. List each table family and exactly where its shape is repeated today.

  • Repository core and facts.
  • Auth and CLI auth.
  • Collaboration rows.
  • Read models.
  • Outbox and cleanup jobs.
2

Move table-family entities out of the catch-all file

Split entities.rs into family-owned modules such as entities/repository.rs, entities/auth.rs, entities/collaboration.rs, entities/jobs.rs, and entities/read_models.rs. This is a mechanical first move that lowers blast radius before deeper deletion. 

db/
  entities/
    mod.rs
    auth.rs
    repositories.rs
    collaboration.rs
    jobs.rs
    read_models.rs
3

Co-locate schema builders with table families

Move table creation for each family next to its entity module or into a matching schema/* family module. The important rule is that a developer changing one table should not have to hunt through one broad schema file and one broad entity file.

4

Shrink drift and reset machinery

Reassess metadata_schema.rs under pre-alpha destructive reset rules. Keep the startup guard that matters, but remove hand maintained table/column inventory if the same outcome can be achieved by detecting coarse drift and resetting incompatible local state.

Bias: because there are no users and no valuable data, compatibility migration machinery should lose to simpler destructive reset behavior.

5

Make the intended edit path obvious

Add a short comment or doc near the DB module entry point that says where a new table, new column, entity conversion, and destructive reset expectation should live. The point is to stop future schema work from re-growing parallel inventories.

Boundaries

This work should make persistence easier to change without changing the domain model's ownership of behavior.

Do Do not
Delete duplicate schema representation where destructive reset is enough. Add backwards-compatible migrations for pre-alpha state.
Split files by table family when it lowers change friction. Normalize the full source graph, visibility events, or staged update state.
Keep domain transition functions authoritative. Move business rules into SeaORM entities or migration helpers.
Preserve current API, Git, worker, and auth behavior. Use this work to sneak in read-path caching or binary-file behavior changes.

Verification

Since this is intended to be behavior-preserving, verification should prove unchanged behavior and reduced edit surface.

  • Run cargo fmt --check.
  • Run cargo clippy --workspace --all-targets.
  • Run cargo test --workspace.
  • Run API contract generation/checks if entity exports change.
  • Run existing DB/schema tests that cover reset and drift behavior.
  • Confirm no file in crates/scope-core/src/db exceeds 1000 lines.
  • Document the before/after count of files touched for a representative column addition.
  • Run autoreview with the repo timeout preference of 15 minutes before closeout.

Next Decisions

The implementation should answer these while editing code, with the simplest defensible choice winning.

Inventory

Can metadata_schema.rs be reduced to a drift/reset guard instead of a full table/column registry?

Authority

Should entity modules own table identifiers directly, or should a tiny shared identifier module remain for SeaORM migration builders?

Granularity

Which split gives the smallest practical modules without making every table its own tiny file?

Closeout rule: stop when the schema edit surface is materially smaller. Do not keep refactoring until the DB layer looks perfect.
