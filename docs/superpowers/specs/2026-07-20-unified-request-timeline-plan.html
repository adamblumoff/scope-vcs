Scope · Request review foundation · 20 July 2026

One request.
One review thread.

Replace the Discussion / Changes split with one stable, chronological review timeline. A push becomes a first-class change block; a change block can own the same nested conversation as any other root post.

Recommendation: build this as a domain-owned timeline, not a browser-side merge. Generalize today’s discussion root into a subject-backed thread and let revision events become commentable subjects.

01 / SHAPE

The decision in one screen

Keep Reddit-style nesting inside each root item, but order roots by when they entered the request. Replies never pull an old root to the bottom. That preserves chronology while still giving focused review conversations a home.

Root type 01Free-standing discussion
Root type 02Collapsed change block from a push
Nested contentReplies attached to either root
Ordering lawRoot position never changes

The key move is not visual. Today the browser chooses between independent resources. The target API returns one ordered union whose source positions are allocated by the request aggregate.

Build the UI ourselves

Reuse Scope’s markdown, composer, reply rendering, badges, and SSE subscription. Reuse the repository History page for diffs. Chat packages optimize for flat, inverted, high-frequency streams; this is a domain-specific review document.

Comment at block level

A v1 comment targets a whole push. Line- and file-level anchors require durable patch-coordinate semantics and can be added after the block identity is stable.

Keep audit history quiet

Discussion and code changes join the main surface. Administrative lifecycle events stay in the existing request-history overflow drawer, preserving the earlier “available, not prominent” decision. Repository commit History remains the independent code inspection surface.

02 / UX

The desktop experience

Remove the tabs entirely. The request description is the opening post, followed by an unbroken review timeline. Change blocks are collapsed by default, but they occupy the same root hierarchy as comments and expose the same reply affordance.

Interaction rules

“New discussion” scrolls to and focuses the single root composer at the bottom. “Comment on this change” opens an inline reply composer under that block. Every code-change block starts collapsed; expanding it reveals only the changed-file manifest. Each file has a “View diff ↗” button that opens the repository History page for that request revision and file in a separate browser tab, leaving the conversation untouched. Expansion is ephemeral local UI state: every block loaded, refreshed, or received through catch-up starts collapsed.

Filtering without destroying context

Default is the full timeline. A compact “Unresolved only” toggle may hide resolved comment roots and change blocks with resolved conversations. Delete “Recently active” and “Newest”: roots always render by ascending root_position.

03 / DOMAIN

Make “commentable” an explicit domain rule

Do not attach comments directly to a UI card or loosely polymorphic URL. A request thread owns conversation behavior; its subject determines what is rendered as the root. The request aggregate allocates every position and performs push + block + thread creation atomically.

A change block is eager, not created on first comment. Its conversation state begins None; the first reply transitions it to Open. Empty blocks therefore do not inflate the unresolved count.

Historical diffs need retained data

The current changes endpoint computes only base_main_oid → current head_oid, and the revision write queues the previous request snapshot for deletion. A real historical change block cannot depend on those objects surviving by accident.

Plan: add a revision-owned snapshot reference to each block and stop orphaning a snapshot while a live block owns it. Hydrate the adjacent snapshots on demand for old_head → new_head. Persist file counts and the path/kind manifest at push time so the collapsed block never needs Git work. Only the request-revision mode of the History page performs snapshot hydration, after a user opens a specific file in its separate tab.

One source of order

Request.activity_version remains the only sequence allocator. Root threads, replies, resolution transitions, and revision blocks consume it through domain behavior. Timestamps are display metadata only; they never determine ordering or cursors.

Revision details live only on RequestChangeBlock. The audit event references the block id instead of repeating heads, stats, or snapshot metadata.

The timeline query pages roots by root_position. A reply changes last_activity_position for unread/catch-up calculations but not root_position.

04 / WEB + API

A smaller route and one bounded read model

The route loader fetches request detail, account, and one latest timeline page in parallel. File manifests and older roots are demand-loaded; file diffs belong to a separate History-page tab. The existing repo SSE stream stays a hint channel; the timeline endpoint remains authoritative.

URL model

Delete ?view=discussion|changes, discussionSort, and the separate changes route state. The request conversation keeps only root anchors such as ?thread=thread_id. File inspection is an explicit link to the existing repository History route in a new tab: /repos/$owner/$repo/history?request=$requestId&revision=$blockId&path=$path. Extend HistorySearch and HistoryPage to load an authorized request revision because today that page only knows projected repository commits.

06 / PROOF

What “solid working version” means

The implementation is done when ordering, durability, permissions, and catch-up are proven—not when the mockup happens to look right on one seeded request.

✓

Atomic push invariant

A successful submitted-request push creates one revision event, one change block, and one subject-backed thread at the same position. A failed write creates none.

✓

Stable chronology

New replies, resolution, unread updates, and realtime catch-up never reorder existing roots. Older-page prepend keeps the user’s viewport anchored.

✓

Durable historical review

After three pushes, the first and second block still show accurate manifests. Every file button opens the correct old-head-to-new-head diff on the repository History page in a separate tab, even after cleanup jobs run.

✓

One conversation behavior

Comment roots and change blocks share reply, quote, unread, resolve, reopen, authorization, idempotency, and realtime behavior.

✓

Bounded first render

The loader returns at most 40 roots and three reply previews per root. Change blocks are collapsed by default. No historical Git hydration or file content occurs when the user expands one; only the separate History tab loads a selected diff.

✓

Contributor → maintainer E2E

Submit, push twice, comment on push one, reply, resolve, push again, expand a collapsed block, open a file diff in a new tab, verify the History page selected the right request revision and path, return to the untouched conversation, refresh, and verify the same chronological timeline.
