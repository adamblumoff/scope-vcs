Shared projection identity

Make Git history, API history, and projection caches canonical by effective view class so owners and full-read members get the same private view instead of user-shaped histories.

Product rule

Same effective view, same identity.

Projection identity should be keyed by what the requester can see, not by who the requester is. Owners and members who can read private files should share the canonical private view. Public readers should share the canonical public view. Future scoped agents can add more view classes, but the key remains the visible repository shape, not the token or user id.

Recommended decision: default Git clone and API history are live audience views. The private live view is shared by owner and full-read members. Review or staged snapshots should not be hidden behind the owner's default clone path.

What exists today

The repo already has the pieces, but they disagree.

Projection domain is principal-shaped

project_graph accepts a Principal and emits Projection.principal_id. Projected commit IDs are built as pv_{principal}_{source}_{seq}, so API history and preview IDs are user-shaped even when content is identical.

Git cache is principal-shaped

projection_cache_key hashes projection.principal_id, commit.projected_id, and parent projected IDs. That creates duplicate bare repo caches for equivalent views.

Owner clone is special

Authenticated owners bypass the synthetic projection and receive the raw Git snapshot when available. Members receive a synthetic projection, even when they can read the same private files.

Read models already point the right way

Projection file read models are already keyed by audience = private|public. That should become the shared concept for Git, history, preview, and file read paths.

Important correction: principal-shaped projected_id is not directly written into the synthetic Git commit object. The more serious SHA divergence is owner raw snapshot versus member synthetic projection. The plan fixes both by making the view class explicit everywhere.

Target architecture

Introduce one canonical projection view key.

Add a domain-owned view identity and make every projection caller use it. The first version only needs two classes:

Implementation sequence

Keep it small and proof-driven.

1

Pin the behavior with failing tests

  • Owner clone token and member clone token for the same full-private repo resolve the same main SHA.
  • Two members with identical private access resolve the same projection cache key and API history projected IDs.
  • Public clone still omits private files and produces a different head when private files exist.
  • Default owner clone does not serve a staged update snapshot unless a future explicit review clone route asks for it.

Delete in this step: remove or rewrite tests that bless owner-only clone behavior as the desired contract. Keep the raw snapshot tests only when they prove the shared private view, not an owner exception.

2

Add a domain projection view key

Create a small domain type such as ProjectionViewKey with private and public. Make project_graph accept the view key and can_read_private_files instead of a principal used only for naming.

Delete in this step: remove projected_principal_id and stop threading Principal through projection code for identity. Keep principal only at the authorization boundary where it actually answers access questions.

3

Make projected IDs audience-stable

Generate projected IDs as pv_{view_key}_{source}_{seq}. Replace Projection.principal_id with Projection.view_key. API responses should expose view_id or view_key, not principal_id.

Delete in this step: remove principal_id from Projection, ProjectionPreviewView, CommitHistoryView, response structs, and the generated TypeScript contract. Do not leave alias fields.

4

Unify Git upload-pack around the view key

Replace owner_snapshot_repo_for_request with a private live repo resolver. Any requester in the private view gets the same live raw snapshot when available. Public requesters get the synthetic public projection. Synthetic projection cache keys hash the view key, not user identity.

Delete in this step: remove the owner-only upload-pack branch and the path where default owner clone serves staged_update.git_snapshot. Review snapshots need an explicit future route if the product wants them.

5

Rename owner audience to private audience

The UI and API should talk about private and public, not owner and public. Owners are one kind of private-view participant, not a projection class.

Delete in this step: remove owner as an API audience value, route search value, UI label branch, and generated TypeScript union member. This is pre-alpha, so do not keep owner compatibility parsing.

6

Centralize audience derivation

Move the duplicated audience-selection helpers from repo reads and projection read models into one domain-owned helper. The DB read model can keep its existing private and public audience strings.

Delete in this step: remove duplicate live_projection_audience and projection_preview_principal-style helpers that independently infer the same audience class.

7

Bump projection cache semantics

Update PROJECTION_CACHE_SEMANTICS_VERSION because cache keys, projected IDs, and private clone behavior change. No migration is needed; old local caches can be abandoned.

Delete in this step: remove stale cache-key assumptions from tests and docs. Do not add cleanup code for old cache directories.

Delete while changing

Remove the user-shaped projection surface.

This should be a subtraction-shaped change. The goal is not to add a second identity layer beside the current principal-shaped one; it is to replace the current identity surface with the view-key model.

Verification

The proof is a real clone, not just API JSON.

Unit tests

  • project_graph emits identical IDs for identical view keys.
  • Projection cache key is identical for two private readers.
  • Public and private view keys produce distinct projections when private content exists.

API tests

  • Private history route returns stable projected IDs for owner and member.
  • Public history continues to redact private commits and versions.
  • Generated TypeScript contract uses private and public.

Git tests

  • Owner and full-read member clone the same repo and get the same rev-parse main.
  • Two full-read members get the same rev-parse main.
  • Public clone omits private blobs and has the expected redacted history.

Manual QA

  • Create a repo with public and private files.
  • Create owner and member clone credentials.
  • Clone as both, compare git rev-parse main, then inspect files.
  • Clone anonymously or as public and verify private files are absent.

Not doing

Keep this slice about identity, not new permissions.

  • No scoped agent token implementation in this PR.
  • No per-path member ACLs yet.
  • No GitHub mirror or branch model work.
  • No compatibility aliases for old owner audience names.
  • No durable migration for old projection caches or old local projected IDs.
Recommended landing shape: one destructive PR that converts projection identity from principal-shaped to view-shaped, unifies owner/member private clone behavior, updates API/UI naming, and proves it with real Git clone tests.
Scope pre-alpha architecture note. This plan intentionally favors a destructive cleanup over compatibility because the product has no users or valuable legacy data.
