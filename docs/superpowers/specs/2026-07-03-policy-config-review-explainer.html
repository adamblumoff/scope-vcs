Scope VCS policy model

Make visibility move forward, not backward.

The config file should decide what happens next. Old commits should keep the visibility they had when they were created, unless the config asks for a destructive rewrite on a specific path.

The simple rule: a commit carries its own public/private label. A later config edit does not silently relabel old commits.

Config file: always private Scope metadata.

CLI: local review and human choice.

Server: final validation and public exposure.

What is wrong today?

Today, the public view can be rebuilt by looking at the current policy. That means changing policy now can change what old history looks like. That is powerful, but surprising.

Today: current policy rewrites the past destructive by default

Commit A

File starts public. Public readers can see it.

Commit B

More public changes are added.

Policy flips

The file is now private.

Old view changes

Public history can lose A and B after the fact.

Proposed: each commit keeps its label forward by default

Commit A

Public then, public forever in public history.

Commit B

Also remains visible because it was public then.

Config flips

From this point forward, the file is private.

Commit C

New private changes stay out of public history.

What happens when visibility changes?

Changing visibility creates a boundary. The boundary affects the public checkout from that point forward, but it does not rewrite older commits unless the config explicitly asks for that.

Public -> Private

Public readers keep old public commits. The public checkout gets a forward delete boundary so the file disappears from the latest public tree.

Private -> Public

Public readers get a new baseline with current content. They do not get the old private history that created that content.

Destructive cleanup

Old public history is removed only when the config names a path and asks for a destructive rewrite.

The config file becomes the write surface.

Instead of letting the web UI and CLI both edit visibility, make the config file the only place that changes file visibility behavior.

Thing Plain meaning
.scope/repo.json Reserved Scope config. It is always private, even if the repo owner tries to make it public.
visibility.rules Normal forward-only public/private decisions for paths.
history.rewrites Explicit destructive operations. They must name the path or directory being rewritten.
Web UI Preview and explain visibility. Do not mutate visibility once config owns that job.

Invariant: the server also enforces the config file privacy rule. This cannot be only a CLI promise. 

{
  "$schema": "https://scope.dev/schemas/repo-config.v1.json",
  "kind": "scope.repo-config",
  "version": 1,
  "visibility": {
    "default": "private",
    "rules": [
      { "path": "/README.md", "visibility": "public" },
      { "path": "/src/**", "visibility": "public" },
      { "path": "/notes/**", "visibility": "private" }
    ]
  },
  "history": {
    "rewrites": [
      {
        "path": "/leaked-secrets/**",
        "action": "redact-public-history"
      }
    ]
  }
}

Why move review into the CLI?

Review is about deciding what this push should expose. That decision is easier when the CLI can inspect local files, local config, and the exact Git change before upload.

1

Read config

The CLI reads the reserved config and computes intended visibility.

2

Compare base

The CLI compares local HEAD to the last Scope-accepted snapshot.

3

Local review

The terminal shows changed files, visibility, and any destructive history requests.

4

Server validates

The server rejects stale bases, invalid config, and attempts to expose reserved private files.

5

Apply atomically

Content, policy boundaries, and optional rewrites land together.

The short version.

Keep the human review local, but keep privacy enforcement on the server. Make normal policy changes forward-only. Make destructive history changes explicit, path-scoped, and config-only.
