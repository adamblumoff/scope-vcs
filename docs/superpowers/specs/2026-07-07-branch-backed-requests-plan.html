Approved product direction

Branch-backed Requests with credit-staked public submissions

Build the Requests system from the branch workflow outward: durable request records, server-owned request refs, CLI-first branch writes, web-visible review timelines, clean web merge, and deterministic credit settlement for public contributors.

The first version intentionally supports only branch-backed Requests. Text-only issues, feature proposals, bounties, ratings, direct credit purchases, maintainer reputation, and browser-based conflict resolution are deferred until the branch and settlement foundation is boring.

Thesis

A Request is the maintainer-facing economic object. In v1 every Request is backed by a branch-like server ref. Public contributors stake credits when submitting; owners and members may create Requests without participating in the credit economy. Public users can never push main. Owners and members can update main, including through a clean web merge action.

Object

Request

One durable domain object owns lifecycle, base projection, request ref, current head, stake, disposition, and settlement facts.

Git

Request ref

Submitted code lives under a server-owned ref such as refs/scope/requests/req_123, not on main.

Economy

Stake once

Public contributors pay stake on first submit only. Updates to the same Request are free until the Request is resolved or withdrawn.

Boundary

No leakage

Public Requests are based on public main. Private files, private history, and private conflicts must not leak to public authors.

Non-Negotiable Rules

These rules drive the data model and should be enforced in scope-core, with HTTP, CLI, and web acting as delivery adapters.

Public users never push main

Public contributors may create and update request refs. Main is updated only by owner/member actions.

Base projection follows auth

scope request start authenticates. Public users start from public main; members and owners start from private main.

Web can mutate workflow

Web may comment, ask for response, resolve, and clean-merge a Request. It is not a general Git push surface.

Clean merge only

Browser merge succeeds only when the backend can cleanly merge the request head into the confirmed current main.

Abandonment requires contributor turn

A contributor can be penalized only after the Request is waiting on them. Maintainer silence cannot burn contributor stake.

Settlement is deterministic

Maintainers pick dispositions. Domain code maps dispositions to refund, reward, partial refund, or burn.

Domain Model

Put the core model under crates/scope-core/src/domain. The API layer should validate transport and auth, then call domain commands. Persistence should store typed request and ledger facts rather than hiding the economy in one mutable user row.

State Meaning Allowed transitions
Submitted Request is open. The author can push new revisions. Maintainers can comment, ask for response, resolve, or merge. NeedsResponse, Resolved, Withdrawn
NeedsResponse Maintainer asked for something specific. Contributor updates or explicit response move it back to Submitted. Submitted, Resolved
Resolved Maintainer selected a disposition. The request ref is locked and settlement is final for v1. None in MVP
Withdrawn Author closed the Request before maintainer resolution. Refund behavior depends on whether review work has happened. None in MVP

Git And Branch Model

The existing staged push review concept is dead for this feature. Requests use separate refs and explicit merge commands. Public contributors pull and rebase against public main only; members and owners pull private main when their auth allows it.

1

Authenticate

CLI resolves repo access and prints whether the user is operating as Public, Member, or Owner.

2

Start

CLI fetches the allowed main projection and creates a local request branch from it.

3

Submit

CLI pushes local HEAD to a request ref and creates the durable Request. Public users pay stake.

4

Update

Author can push new revisions to the same request ref without additional stake.

5

Review

CLI and web show request diff, timeline, stake, state, and next actor. Comments can happen in both places.

6

Merge

Owner/member confirms clean web merge into current main or uses CLI for local conflict resolution. 

Ref rules
  Public author:
    can create/update refs/scope/requests/{own_request_id}
    cannot update refs/heads/main
    base_audience = Public

  Member/owner author:
    can create/update request refs with stake_credits = 0
    can update refs/heads/main only through owner/member permissions
    base_audience = Private

  Resolved/withdrawn Request:
    request ref is locked from further author updates

API Surface

Keep API endpoints thin. They should identify the user, load repo access, validate request payloads, invoke domain commands, persist typed facts, and return generated TypeScript response contracts.

Endpoint Used by Purpose
GET /v1/repos/{owner}/{repo}/requests Web, CLI List request summaries visible to the current principal.
GET /v1/repos/{owner}/{repo}/requests/{id} Web, CLI Get request detail, mergeability, settlement preview, and timeline.
POST /v1/repos/{owner}/{repo}/requests CLI Create a submitted branch-backed Request after request ref push.
POST /v1/repos/{owner}/{repo}/requests/{id}/revisions CLI Record a pushed update to the request ref.
POST /v1/repos/{owner}/{repo}/requests/{id}/comments Web, CLI Add a timeline comment without changing lifecycle state.
POST /v1/repos/{owner}/{repo}/requests/{id}/needs-response Web, CLI Maintainer asks for contributor action with a required note.
POST /v1/repos/{owner}/{repo}/requests/{id}/respond Web, CLI Contributor replies and returns the Request to Submitted.
POST /v1/repos/{owner}/{repo}/requests/{id}/resolve Web, CLI Maintainer resolves with a non-merge disposition and settlement.
POST /v1/repos/{owner}/{repo}/requests/{id}/merge Web, CLI Owner/member clean-merges into confirmed current main, resolves as Accepted, settles.

CLI UX

The CLI must be explicit about auth and projection every time it starts, submits, syncs, or updates a Request. Users should never have to infer whether they are using public main or private main.

scope request start

Authenticates, prints access level, fetches allowed main projection, creates local branch, and explains what commands are valid for that user.

scope request submit

Shows diff summary, stake cost, current balance, target projection, and pushes request head. Public users pay stake once.

scope request sync

Pulls the latest allowed main projection into the local request branch. Public users sync from public main only.

Web UX

The web UI can mutate request workflow and perform clean merges. It should not become a general Git editor. The request page is a maintainer queue, diff viewer, timeline, settlement preview, and confirmation surface.

Request list

Filter by open, needs response, resolved, author, and mergeable. Show stake only when relevant to the viewer.

Request detail

Show title, state, base audience, current head, changed files, timeline, comments, and available actions.

Merge dialog

Confirm exact current main OID, request head OID, target branch, disposition, and settlement before backend merge.

Stale merge handling

If main changes after the dialog opens, reject and require refresh before merge. 

Merge confirmation copy
  Merge req_123 into main?

  Current main: abc123
  Request head: def456
  Settlement: refund 10 credits, reward 5 credits

  This will update main and resolve the Request as Accepted.

Settlement Logic

Settlement runs once, after resolution. For accepted code Requests, settlement happens only after merge succeeds. Maintainer dispositions are product judgments; credit outcomes are domain-owned rules.

Disposition When used Default settlement
Accepted Clean merge succeeded and main was updated. Refund full stake plus reward 50% of stake.
UsefulNotMerged Work was helpful but maintainers chose another path. Refund full stake plus reward 20% of stake.
HiddenContext Public contributor could not reasonably know why it could not land. Refund full stake. No penalty.
NotAligned Good-faith work that is not a project priority. Refund full stake for MVP.
Duplicate Contributor missed visible public context. Refund 50% of stake. Burn the rest.
Abandoned Only after NeedsResponse and contributor deadline passes. Refund 0% in MVP unless manually resolved otherwise.
LowQuality Low-signal, misleading, spammy, or no meaningful diff. Refund 0%. Burn full stake.

Deletion Track

This project should remove the old server-side staged-update review concept as branch-backed Requests come online. The CLI visibility review TUI remains useful and should not be deleted merely because it uses the word review. The dead thing is the server-side pending staged update queue as a product workflow.

Phase Delete or replace Keep or rename
Phase 1 Do not add compatibility shims for staged update review. New request and ledger domain code should stand on its own instead of wrapping StagedRepoUpdate. Keep generic file-change concepts only if they are renamed away from staged-update ownership.
Phase 2 Replace receive-pack paths that create server pending staged updates with request-ref admission and request revision recording. Remove public/main write paths that only existed to feed server staged review. Keep lower-level Git import, snapshot, and diff helpers when they can serve request refs without preserving the old workflow.
Phase 3 Remove staged-update response fields from repo summaries, especially staged_update_pending and push_blocked_by_staged_update. Remove generated TypeScript exports that exist only for staged update review. Replace them with request counts, request permissions, and mergeability fields owned by the Request read model.
Phase 4 Remove CLI messages and guards about stale staged update state. Delete API client fields that mirror staged-update repo summary flags. Keep scope review and the push visibility review TUI as local config/diff tooling. Rename only if the new command vocabulary makes the old labels confusing.
Phase 5 Remove web assumptions that repo state is blocked by a single staged update. Delete any stale request/review UI code that cannot represent multiple branch-backed Requests. Keep generic diff drawer and file-tree UI if it can be reused for request diffs or history without staged-update language.
Phase 6 Delete StoredRepository.staged_update, StagedRepoUpdate, StagedFileChange, staged update apply/reject commands, staged update DB JSON columns, schema contract entries, and tests that only protect the removed workflow. Preserve behavior by moving any still-needed apply/diff logic into Request-owned modules before deleting the staged names.

Deletion should happen as soon as the replacement slice is in place, not only at the end. The final closeout phase should be a confirmation pass, not the first time dead staged-review code is touched.

Deferred

These are valid product layers, but adding them now would muddy the foundation. The MVP proves branch-backed requests, public contribution staking, clean maintainer merge, and deterministic settlement.

  • Text-only requests for bugs, questions, design proposals, and bounties.
  • Visible contributor or maintainer ratings.
  • Direct credit purchases, pricing tiers, or public/private pricing spread.
  • Browser conflict resolution.
  • Maintainer punishment logic for slow review. Track facts first.
  • Complex repo policy engines for request pricing or submission rules.

This plan is intentionally destructive and pre-alpha native. It does not preserve staged update review as a product concept. Requests become the branch-backed maintainer attention primitive.
