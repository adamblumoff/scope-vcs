Implementation plan

Make visibility review a local TUI, not a JSON editing chore.

Add scope review as the canonical way to edit .scope/repo.json, and make scope push run that review by default unless --no-review is passed.

The server and web app should continue treating the repo config as the only visibility source of truth. The new work is local CLI ergonomics: a collapsible file tree with keyboard visibility toggles that writes the same config file the backend already validates.

Settled rule: if review changes .scope/repo.json during scope push, save the config and stop. The user commits that config, then reruns scope push. No hidden auto-commits.

Target behavior

The TUI should feel like the old folder/file visibility tree, but it writes local config instead of mutating server state.

Standalone command

scope review opens a file tree for the current Git worktree and writes .scope/repo.json. It can be used before a commit to prepare visibility policy.

Push integration

scope push opens review before creating the push intent. scope push --no-review skips the TUI and pushes every committed file with the committed config.

Config stays authoritative

The TUI only edits .scope/repo.json. The server still validates that config on receive and applies visibility from the committed config.

TUI shape

The interaction model is intentionally small: move, expand, toggle, save, continue, quit. This first pass edits visibility only.

Keyboard rules

Folder behavior should be predictable and fast. A mixed folder becomes a concrete subtree rule when toggled.

Input Behavior Config effect
Space on file Toggle public/private for that file. Create or update a rule like /README.md.
Space on folder Toggle the whole subtree public/private. Create or update a rule like /src/**.
Right on folder Expand the folder. If already expanded, move to first child. No config change.
Left on folder Collapse the folder. If already collapsed, move to parent. No config change.
S Save the current policy to .scope/repo.json. Write canonical formatted JSON.
P during push Continue the push if config is unchanged and committed. If config changed, save and stop for commit.
Q or Esc Cancel review. No write unless the user explicitly saved first.

Push flow

The push flow should run review before creating the short-lived push intent, so users can take time in the TUI without expiring the server credential.

1

Prepare repo

Verify Git repo and HEAD. Warn if the worktree is dirty, because only committed HEAD is pushed.

2

Run review

Unless --no-review, open the TUI using the committed HEAD tree and changed-path context. With --no-review, skip review and make no local config edits.

3

Stop if edited

If review writes config, stop with: commit .scope/repo.json, then rerun scope push.

4

Load config

Require committed config and parse it at the reviewed HEAD, not from the mutable worktree. Every committed file is evaluated against explicit rules first, then visibility.default.

5

Create intent

Create push intent after review, then recheck base state against the intent response.

6

Push

Push the reviewed commit to refs/heads/main with the intent credential.

Config editing rules

The TUI should produce simple, predictable config. It should not try to preserve hand formatting or comments because the file is JSON.

Canonical rules

Every save rewrites .scope/repo.json in a stable order. That makes review diffs readable.

Case Rule emitted
Root toggle Change visibility.default.
Folder toggle { "path": "/folder/**", "visibility": "public" }
File toggle { "path": "/folder/file.ts", "visibility": "private" }
Rule equals inherited value Remove the redundant rule.
Parent subtree rule replaces children Remove child rules only when they now equal the parent effective visibility. Keep child overrides that intentionally differ.
/.scope/** Never emit public rules. Always display forced private.
--no-review with new files Apply the committed config to all committed files. Added files use the most specific matching rule, or visibility.default if none matches.

Tradeoffs

Better local control

Review stays local and fast. We avoid syncing an editable review state between web, API, and CLI.

Push can become two-step

If the TUI edits config during push, the user must commit that config and rerun push. This is intentional commit-forward policy.

Terminal complexity

Ratatui/crossterm add dependency and QA surface. Keeping the state model pure limits the risk.

Locked decisions

Push-time edits: save .scope/repo.json and stop. The CLI does not auto-commit, amend, or push an uncommitted config change.

Review skip flag: delete -y/--yes. scope push --no-review skips only the TUI and pushes every committed file using committed config rules plus visibility.default.

TUI scope: visibility only. Destructive policy changes get their own CLI command later because they should be rare and explicit.

Plan written for the current config-owned policy architecture: .scope/repo.json is private, committed, and validated by Scope on push.
