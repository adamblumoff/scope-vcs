Outcome

One behavior, one owner, fewer lines.

This is an architecture reduction campaign, not a formatting pass. It removes unreachable product surfaces and parallel implementations before reorganizing what remains. File splitting alone does not satisfy the plan; code moved from one file to another contributes zero to the reduction.

Target a net deletion of 8,000–10,000 hand-owned lines while retaining API behavior, authorization semantics, Git protocol behavior, CLI flows, request settlement outcomes, and visible web behavior.
Dimension Required result
Line reduction At least 7,000 net lines; expected 8,000–10,000.
Behavior Existing supported workflows and externally observable contracts remain unchanged.
Architecture Domain rules are delivery-independent; persistence has one production implementation.
Tests Regression value is preserved or improved; duplicate mechanics and fixtures are consolidated.
Operations External I/O no longer runs while a global metadata transaction is held.

Operating rules

Reduction without games.

Delete vertically. Remove model, persistence, API, CLI, UI, and tests together.

Keep one source of truth. Consolidate rules before deleting their copies.

Do not minify. Compressed, generated, macro-hidden, or unreadable code does not count.

Do not weaken coverage. Tests count only when redundant mechanics are removed.

No compatibility layer. This is pre-alpha; destructive schema resets are preferred.

Stop on behavior change. Any intended product change requires owner approval.

Prefer subtraction. New abstractions must delete more machinery than they add.

Verify each phase. Do not stack a later cleanup on an unverified earlier phase.

The LOC target is a constraint on architecture quality, not permission to erase meaningful tests, combine unrelated responsibilities, or replace clear code with clever code.

Execution phases

Delete in behavior-owned slices.

Freeze behavior and establish the ledger

Prove what must remain before deleting its current implementation.

+0 to +200 Net line budget

Work

  • Record supported route, DTO, CLI, Git protocol, visibility, request-transition, settlement, and SSE behavior.

1

Delete unreachable product surfaces

Remove concepts that production cannot create or consume.

−1,100 to −1,800 Expected net deletion

Pending-import vertical deletion

  • Remove PendingImport, PendingImportFile, and StoredRepository.pending_import.
  • Remove persistence columns, entity conversion, publish/apply paths, read-model booleans, API/CLI DTO fields, web branching, and tests that manufacture the state.
  • Keep the currently supported direct first-push publication flow.

Unowned backend settings deletion

  • Remove backend RepoSettings, its table/entity/read/update modules, and tests for the nonexistent endpoint.
  • Retain repository collaboration/settings UI; it is a separate live feature.
  • Keep default visibility in RepoConfig, its actual owner.

Small deletions

  • Delete obsolete-route tombstone tests, stale suppressions, dead CLI Git helpers, stale React Doctor configuration, and verified-unused dependencies.
  • Simplify redundant feature aliases and duplicate script names.

Exit gate

No production or test reference remains to the deleted domain concepts, schema resets cleanly, and supported first-push/settings workflows pass.

2

Collapse persistence to one implementation

Make Postgres the sole metadata runtime and remove catalog-era machinery.

−3,500 to −4,500 Expected net deletion

Postgres only

  • Delete memory-metadata, MemoryMetadataStore, and every memory/Postgres match branch.
  • Run local development and integration tests against disposable Postgres.
  • Keep pure domain tests on in-memory domain values without pretending they are a second database.

Native async API

  • Make MetadataStore methods async and await SeaORM on the server runtime.
  • Delete DbRuntime, run_api_db_on, run_db_on, the blocking sync channel, and leaked runtime.

Focused persistence

  • Delete production MetadataStore::read and full-catalog hydration.
  • Add focused queue, collaboration, user, request, repository, and blob-reference reads.
  • Replace the global catalog lock with affected-row or aggregate locks and database constraints.
  • Persist mutation-owned deltas instead of rewriting the whole repository aggregate.

Schema ownership

  • Choose one schema source; remove duplicated schema-contract machinery and unreachable preservation branches.
  • Use destructive pre-alpha reset rather than compatibility migration code.
  • Replace saturating row/domain conversion with checked fallible conversion.

Exit gate

Local dev, API, worker, and tests use Postgres; no synchronous DB bridge or full-catalog production read remains; benchmark results do not regress materially.

3

Give policy and contracts one owner

Make adapters translate decisions instead of reimplementing them.

−500 to −900 Expected net deletion

Domain independence

  • Replace Axum-coupled ApiError in scope-core with typed domain/application errors.
  • Map errors to status codes, public messages, and diagnostic sources only in the API boundary.
  • Structurally enforce the crate boundary so the handwritten source lexer can later be deleted.

Policy consolidation

  • Create domain-owned request visibility, actor role, permissions, mergeability, and push-authorization decisions.
  • Move repository discoverability/readability to one domain capability owner and delete API/catalog copies.
  • Return allowed resolution options and settlement previews computed by the domain; delete TypeScript math.

Transport contract

  • Create a small API transport-contract crate shared by API and CLI; keep it separate from durable domain code.
  • Generate TypeScript from the shared contract and preserve the existing stale-contract check.
  • Add a six-variant resolution disposition, a typed Git OID, typed projection persistence encodings, and a generated SSE payload.
  • Centralize route templates and path-parameter encoding.

Exit gate

Authorization and settlement decision tables have exactly one executable implementation, scope-core has no Axum dependency, and API/CLI/web contracts are drift-checked.

4

Simplify side effects, Git, and CLI orchestration

Replace manual cleanup and duplicate workflow owners with explicit lifecycles.

−800 to −1,200 Expected net deletion

External side effects

  • Change cleanup to claim in a short transaction, perform filesystem/S3 work outside it, then generation-check and reconcile.
  • Either implement due-time filtering/backoff/bounds for cleanup jobs or delete unused retry fields.
  • Require outbox completion to verify lease ownership and affected-row count.

Git receive

  • Wrap receive-pack staging repositories in one RAII cleanup guard.
  • Remove repeated remove_dir_all branches and extract shared upload/base-validation/persist/rollback choreography.
  • Keep bounded stderr, timeout, process-group kill, and child-wait behavior.

CLI

  • Delete the unused alternate push-intent workflow.
  • Thin bin/scope.rs into parse-and-dispatch with behavior-owned init, push, and login modules.
  • Preflight once into a prepared request rather than dispatching every request command twice.
  • Unify remote URL parsing, Git command/auth plans, header injection, execution, and temporary bundle ownership.
  • Use Git's config command and locking instead of appending raw config syntax.

Exit gate

Temp resources are owned by guards, external I/O is outside metadata transactions, one CLI push flow remains, and Git/auth security tests pass.

5

Remove competing frontend state owners

Let routes own server state and focused components own local interaction state.

−600 to −900 Expected net deletion

Immediate correction

  • Remove the settings child live-refresh hook; the parent repository layout owns one SSE connection.
  • Delete the no-op canUseRepoLiveRefresh seam and its no-op test.

Data ownership

  • Choose route loader/search state as the single history owner and delete the parallel client fetch/reducer synchronization path.
  • Use loader data as repository collaboration source; mutate, then invalidate. Do not maintain a shadow copy unless a single explicit optimistic reducer is justified.
  • Move repository navigation/access derivation to the parent layout.

Component reduction

  • Split history and request detail only along behavior boundaries while deleting mega-prop and render-phase synchronization code.
  • Replace render-callback slots with direct ReactNode props.
  • Consolidate exact formatting/status helpers and derive badge tone types from the badge component.

Exit gate

One SSE connection and one server-data owner exist per route tree, browser smoke tests pass, and component splits produce net deletion rather than file shuffling.

6

Consolidate test and tooling machinery

Preserve regression protection while deleting copied harnesses and duplicated automation.

−700 to −1,000 Expected net deletion

Test ownership

  • Move durable domain behavior tests to scope-core and import it directly.
  • Move API workflow tests out of api/src behind a narrow public integration-test harness.
  • Delete the handwritten Rust source lexer after crate dependencies enforce the boundary.
  • Replace copied/manual temporary-directory and Git fixture code with shared RAII fixtures.
  • Keep real Git protocol, binary roundtrip, projection privacy, auth, request-transition, and CLI boundary tests.
  • Add only a small Playwright smoke layer for the highest-value web workflows.

CI and deployment

  • Generate the CLI build matrix and expected artifacts from targets.json.
  • Run the web production build on pull requests and make CI invoke the canonical pnpm check.
  • Remove duplicated Railway upload trees and redundant toolchain version declarations.
  • Combine repeated workspace Rust checks where deployment dependencies permit.

Exit gate

The full suite remains green, critical test scenarios remain discoverable, CI has one owner per check/target list, and final LOC is measured from the same baseline command.

Risk and rollback

Sequence makes the campaign reversible.

Risk Control
Deleting behavior disguised as dead code Prove no production constructor/caller, then retain supported behavior characterization tests.
Memory/Postgres semantic drift during removal Make Postgres tests authoritative before deleting memory branches; compare ordering and error behavior.
Async conversion spreads too broadly Convert one persistence behavior family at a time; adapters await directly without sync wrappers.
Authorization drift Decision-table tests precede consolidation; API concealment tests remain at the boundary.
LOC pressure weakens readability Reject minification, giant files, hidden generated logic, or abstractions whose only purpose is reducing count.
Schema incompatibility Use explicit destructive pre-alpha reset; do not add compatibility migrations.
Each phase should land independently. If a phase changes supported behavior, stop and ask the owner rather than carrying an unapproved change into later refactors. Rollback is phase-level revert plus disposable database reset.

Preserve

Good foundations stay.

Pattern Decision
RepoMutation / RepoEffects Keep Expand the explicit-side-effect pattern to notifications and cleanup receipts.
Generated Rust → TypeScript contract check Keep Extend it to the CLI/shared contract and SSE payload.
Durable cleanup/outbox rows Keep Repair claiming, retry, and ownership rather than returning to fire-and-forget work.
Git subprocess safety Keep Preserve bounded stderr, timeout, process-group termination, wait, and redaction.
Pure file-tree and diff parsing models Keep They already have focused ownership and tests.
Cross-platform CLI distribution Keep Consolidate metadata; do not reduce supported targets.

Campaign closeout

Definition of done.

  1. The headline metric is 59,594 lines or fewer; 58,500 or fewer is the expected result.
  2. The full verification matrix is green, including local Postgres development and browser smoke tests.
  3. No pending-import or backend repository-settings concept remains.
  4. No memory metadata backend, synchronous DB runtime bridge, or production full-catalog read remains.
  5. Domain code has no Axum dependency and policy decision tables have one implementation.
  6. External filesystem/S3 work occurs outside metadata transactions.
  7. API, CLI, and web contracts are derived or drift-checked from one transport owner.
  8. Frontend routes have one server-state owner and repository layout has one SSE subscription.
  9. The final summary reports deleted, added, net, and cumulative lines per phase.
Expected finish: 56,500–58,500 hand-owned lines, approximately 12–15% smaller than the July 9 baseline, with unchanged supported behavior and materially clearer domain, persistence, adapter, and test ownership.
