Scope · engineering plan
Implementation plan · 14 July 2026

Make large pushes proportional to the delta.

Replace repository-wide work in scope push with a durable incremental Git path, delta-owned domain behavior, and asynchronous derived work. First push pays for the repository once; later pushes pay for changed objects and paths.

01

Why the current path slows down

Git already sends an incremental pack. Scope then turns that efficient delta into repeated whole-repository work before it returns success.

01

Full worktree and review scans

The CLI scans every untracked path, always fetches Scope main, and builds review state from the complete tracked tree.

02

Buffered Git transport

The API collects the complete receive-pack body before starting Git, so upload and unpacking cannot overlap and memory scales with pack size.

Two full temporary clones

Receive-pack advertisement and POST each clone the current bare repository with --no-hardlinks, then delete it.

04

Every live blob is materialized

Import runs cat-file over the complete pushed tree before it compares OID, mode, and size and discards unchanged files.

live bytes
05

Full Git bundle on every push

Scope repacks all reachable history, reads it into memory, encrypts it, and uploads the whole bundle under a new object key.

The repository graph is repeatedly loaded, cloned, replayed into a live tree, serialized twice, compared, and written as one value.

Referenced new blobs are queued one row at a time, then push synchronously scans every repository graph to decide what can be deleted.

global data

The post-push API downloads and restores the new full bundle to rediscover its head, then performs a usually no-op aggregate mutation.

02

Target push flow

The synchronous path ends after durability and the domain transaction. Everything derived is represented by an outbox job and may safely retry.

1 · Push contextOne lightweight read returns access, config hash, current head, and cache identity.
2 · ReviewFetch only when the remote head changed. Review changed paths; expand the full tree on demand.
3 · Stream receive-packPipe the bounded request into Git quarantine while the client is uploading.
4 · Persist segmentUpload the incoming pack segment once with base/head/integrity metadata.
5 · Apply domain deltaValidate intent and commit head, logical commit, file changes, live rows, refs, and outbox atomically.
6 · Return, then deriveCLI succeeds. Worker updates projections, compacts segments, and collects orphaned objects.

Accent = synchronous durability boundary. Olive = asynchronous, retryable work.

04

Architectural contract

Keep policy and transition rules in durable domain code. Git HTTP, S3, Postgres, the CLI, and the projection worker remain thin delivery mechanisms around that behavior.

Must remain true

  • The push intent binds repository, user, reviewed head, base head, base config, and proposed config.
  • Only a fast-forward update to refs/heads/main is accepted.
  • Git bytes are durable before database rows reference them.
  • The head, logical commit, changed paths, visibility events, object references, and outbox entry commit together.
  • Visibility changes and history rewrites use the existing domain permission rules.
  • A failed transaction leaves an orphan segment that cleanup may safely delete; it never advances the visible head.
  • Public/private projection tables remain derived and may report NotReady until the worker catches up.

Must be removed

  • Full Git bundle creation on every push.
  • Full receive-pack body buffering and duplicate temporary clones.
  • Complete-tree blob materialization for incremental pushes.
  • Repository graph and visibility history stored as growing JSON columns.
  • Inline object cleanup and repository-wide reference discovery.
  • The post-push completion endpoint and its bundle restore.
  • Legacy snapshot readers, dual writes, migration shims, or fallback branches.

09

Verification gates

  • Domain unit tests cover first push, fast-forward update, config change, permission denial, history rewrite, and stale base.
  • Git HTTP integration tests use real git push with streamed bodies, binary blobs, many files, and compressed requests.
  • Crash tests stop after segment upload, after transaction commit, and before response; restart proves recovery behavior.
  • Concurrency tests race two push intents against one base and prove exactly one head advances.
  • Authorization tests prove raw private Git objects are never reachable through public upload-pack or file endpoints.
  • Projection tests prove NotReady is explicit, latest-version jobs win, and retries converge without blocking push.
  • Cleanup tests prove referenced segments survive, orphaned segments are reclaimed, and no request path scans all repositories.
  • Run cargo fmt --all -- --check, cargo test --workspace, CLI tests, web checks, and the benchmark matrix.
10

Risks and mitigations

Database head advances before Git bytes are durable

High

Upload and verify the segment first. The transaction references an immutable object key; on failure, leave an orphan cleanup job and do not advance the head.

Shared raw cache exposes private objects

High

Permissioned raw repositories never back public upload-pack directly. Public reads use derived public projections with tests that attempt explicit private OID wants.

Segment chains make cold restore slower

Medium

Compact asynchronously after a bounded segment count or byte threshold. Manifests make compaction replaceable without changing accepted domain history.

Normalized transaction becomes too wide

Medium

Use batch inserts/upserts for changed paths, keep the repository aggregate lock, and benchmark 1,000-file deltas before finalizing batch sizes.

Worker lag leaves projections unavailable

Medium

Keep explicit generation-aware NotReady responses, coalesce obsolete jobs, expose queue age, and alert on failed latest-version rebuilds.

11

Locked architectural decisions

These choices are approved. Slice 0 tunes thresholds and budgets; it does not reopen the storage or acknowledgement model.

Durable raw Git store

A persistent volume is fast but couples correctness to one runtime. Object storage is already the durable boundary and supports immutable segments.

Chosen

S3-backed immutable pack segments are authoritative; the local bare repository is a disposable cache.

Push success contract

Waiting for projection and garbage collection makes latency unpredictable and mixes derived work into the write contract.

Chosen

Promise durable Git bytes plus committed domain state. Repository browsing may briefly return generation-aware NotReady.

File content source

Keeping a random S3 object per file duplicates the incoming Git pack and creates thousands of first-push requests.

Chosen

Reference Git blobs by OID/mode/size in domain rows and read them from the restored raw cache; do not maintain a second per-file blob store.

Implementation order. Land measurement first, then remove accidental work. Use benchmark evidence to tune limits and compaction thresholds without reopening the approved architecture.

Rollback. Because the product has no valuable data, rollback means redeploy the previous commit and reset the pre-alpha database/object store—never keep two storage models alive.
