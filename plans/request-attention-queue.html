Scope VCS · implementation plan · request attention queue

Requests become a quality signal, not an inbox firehose.

Replace the current submitted / needs-response / disposition system with one durable workflow shared by the web app, API, Git transport, and CLI. Working requests stay quiet. Ready requests become public, staked, ranked work. Completed requests carry one final assessment.

Recommended delivery: six stacked, green PRs. Reset the vocabulary and database destructively first; then land credit transactions, access control, CLI, queue UI, and detail controls. This is pre-alpha: there is no legacy migration path and no compatibility layer.

01

Locked product decisions

These are implementation constraints, not questions to reopen while coding.

Vocabulary

Only Working, Ready for review, and Completed are request states. “On hold” is a Ready flag. Accepted / Neutral / Rejected is the immutable maintainer assessment.

Publication

The first Ready transition permanently publishes a public request. Returning to Working hides nothing that was already published; it only removes the request from triage.

Draft privacy

A never-published Working request is absent from public and maintainer queues. Author and invitees work normally; maintainers retain exact-link moderation access but drafts are not surfaced or counted.

Invitations

Public requests only, immediate by exact Scope handle, at most 30. They grant branch-push permission—not identity delegation, title editing, assessment, or maintainer powers. No notifications.

Limits

Non-maintainers may have at most three Working and three Ready public requests per repository. Maintainers are unlimited and never use credits.

Review invalidation

Any branch update returns Ready to Working, clears hold, and refunds the current stake atomically. The author must inspect the new package and explicitly mark it Ready again.

Completion

Only maintainers assess. Assessment atomically completes and settles once. Merge implies Accepted. Accepted work may complete without merging and may be merged later without another credit effect.

02

One state machine, everywhere

The domain owns every transition and side effect. API handlers, the web app, CLI, and Git receive-pack only ask the domain to perform behavior.

Hold never creates a fourth state. A maintainer push while held still follows the same invalidation path: Working + refund + hold cleared.

03

Visibility and authority are separate

Reading, discussing, pushing, governing, and assessing are distinct permissions. The current “authenticated public user can push” shortcut is removed.

04

Credits create the queue order

The author chooses confidence. The maintainer sees the signal. Ledger entries make every debit, refund, reward, and burn auditable.

Leaving review voluntarily, a branch push, or “Request changes” refunds in the same transaction that returns the request to Working. Settlement can run only once.

Replace resolve.

Maintainer only; Rejected requires --message. Prints exact settlement.

merge
Keep, redefine completion.

Ready merge assesses Accepted atomically. Later merge of Accepted completed work has no credit effect.

close
Replace delete/withdraw wording.

Author closes only from Working. Never-published work is hard-deleted; previously published work becomes Completed without assessment and stays in public history.

08

Architecture: behavior in the center

The plan keeps one source of truth for rules, one atomic persistence boundary for side effects, and thin delivery layers.

Proposed HTTP contract

POST …/ready
Enter review.

{ stake_credits }; enforces cap, balance, author, no hold, publication, debit.

POST …/working
Author leaves review.

Refunds current stake and clears ready timestamp; forbidden while held.

POST …/request-changes
Maintainer leaves review.

Same domain transition, allowed while held, message optional.

PUT/DELETE …/invitees
Add/remove exact user.

Body uses handle for add; response returns canonical user summary and permissions.

DELETE …/invitees/me
Invitee leaves.

Never lets author remove self or silently transfer authorship.

PUT/DELETE …/hold
Toggle group hold.

Maintainer only; idempotent response, no notification side effect.

POST …/assessment
Complete.

{ outcome, body? }; validates rejection body and settles in one transaction.

POST …/merge
Merge.

Ready → Accepted + Completed + merge, or already-Accepted Completed → merge only.

11

Verification and invariants

Tests protect money, privacy, and Git authorization first; UI snapshots alone are insufficient.

All domain transitions table-tested for actor × state × hold × publication
Stake debit/refund/reward/burn and state change commit or roll back together
Two concurrent Ready/assessment/invite operations cannot exceed caps or settle twice
Never-published drafts cannot leak through list, count, search, activity, or Git fetch
Any branch update from Ready returns Working, refunds once, and clears hold
Only author/invitee/maintainer can push; hold blocks author and invitee
Rejected assessment rejects empty/whitespace justification
Merge from Ready completes Accepted once; later accepted merge pays nothing
Ready sort is stake DESC, ready_at ASC, ID ASC across pagination
Maintainer requests are unlimited and never create credit ledger effects
CLI output matches API state and shows exact credit consequence
Browser tests cover desktop/mobile layout, keyboard focus, dialogs, and held locks

Merge gate for every PR

Rust

cargo fmt --check, workspace clippy, unit tests, DB/API workflow tests, and targeted Git receive-pack tests.

CLI

Command parsing, confirmation, request-context discovery, server error rendering, and end-to-end request workflows.

Web

Type generation clean, lint/typecheck/unit tests, then browser verification in the running app at desktop and mobile widths.

Closeout

Run a dead-code/search audit for every removed state, route, permission, command, and disposition before declaring the stack complete.

12

Risks that change implementation

High
Credit/state split-brain
Process crashes after a refund but before state update.
One transaction, aggregate locks, deterministic ledger IDs, uniqueness constraints, and retry tests.
High
Draft privacy leak
Counts, search, or Git upload-pack expose hidden Working requests.
Central visibility predicate reused by projections and Git reads; negative integration tests for every surface.
High
Authorization race
Invitee removal or hold races an in-flight Git push.
Re-check authority inside the ref mutation transaction/CAS boundary, not only at advertisement time.
Medium
Section pagination drift
Stake/order changes duplicate or skip rows.
Cursor contains sort tuple and snapshot/version semantics; stable ID tie-breaker; projection tests.
Medium
Large PR 1
Destructive vocabulary touches all packages.
Keep its review question narrow, use generated-type checks, and stack later UX work instead of mixing polish into the reset.
