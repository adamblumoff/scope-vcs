**The SHA-divergence problem is your existential product question.** Projected commit IDs are keyed per-principal (`pv_{principal_id}_{commit}_{seq}` in `projection.rs`), so two members with *identical* access still get different Git histories and different SHAs. That breaks the most basic collaboration primitive: "look at commit abc123." Consider keying projections by visibility-equivalence class instead of principal — everyone with the same effective view shares one projection. That's simultaneously a product win (shareable SHAs, working PR discussions), a performance win (shared projection cache), and a cost win (one bare repo per class instead of per principal). I'd rank this above everything else.
