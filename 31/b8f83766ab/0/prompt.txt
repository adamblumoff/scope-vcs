**The solution** is to decompress request bodies when `Content-Encoding: gzip` is present, on both Git POST endpoints (`git_upload_pack_rpc` and the receive-pack handler — clients may gzip either). Two ways to do it:

1. **Handler-level (my recommendation):** after `to_bytes`, check the request's `Content-Encoding` header; if it's `gzip`, decompress with `flate2::read::GzDecoder` before passing along. A small shared helper covers both handlers, it's explicit at the Git boundary where the protocol quirk lives, and you can enforce the size cap on *decompressed* output so a tiny gzip bomb can't blow past `MAX_UPLOAD_PACK_BYTES`.
2. **Middleware-level:** tower-http's `RequestDecompressionLayer` on the Git routes. Less code, but it applies decompression semantics route-wide and its size-limit interaction is less explicit than doing it by hand at this boundary.

Here's exactly how I'd do it.

**1. Add the dependency.** `flate2` in `api/Cargo.toml` (it's almost certainly already in your tree transitively via git-related crates, so this costs nothing at build time).

**2. Write one decoding helper in `api/src/git/mod.rs`.** Something like:

```rust
fn decode_git_request_body(headers: &HeaderMap, body: Bytes, max_bytes: usize) -> Result<Vec<u8>, ApiError> {
    match headers.get(CONTENT_ENCODING).map(|v| v.to_str()) {
        None => Ok(body.to_vec()),
        Some(Ok("gzip")) => {
            let mut decoded = Vec::new();
            GzDecoder::new(body.as_ref())
                .take(max_bytes as u64 + 1)
                .read_to_end(&mut decoded)
                .map_err(|e| ApiError::bad_request(format!("invalid gzip request body: {e}")))?;
            if decoded.len() > max_bytes {
                return Err(ApiError::bad_request("git request body is too large"));
            }
            Ok(decoded)
        }
        Some(_) => Err(ApiError::bad_request("unsupported content-encoding")),
    }
}
```

The `take(max + 1)` is the important detail: it bounds the *decompressed* size during streaming, so a 1 KB gzip bomb can't expand to gigabytes in memory before you check the length. The existing `to_bytes(..., MAX_*_BYTES)` cap stays as-is for the wire size.

**3. Call it from both POST handlers.** In `git_upload_pack_rpc` (`api/src/git/mod.rs` around line 160) and the receive-pack handler above it (around line 122), insert the helper between `to_bytes` and the downstream call, passing `MAX_UPLOAD_PACK_BYTES` / `MAX_RECEIVE_PACK_BYTES` respectively. Errors go through the existing `git_upload_pack_error` / `git_error_response` paths so the client always gets a proper ERR pkt-line rather than a bare 4xx.

One design note on why handler-level rather than `RequestDecompressionLayer`: the info/refs GET and the JSON API don't need request decompression, the layer's decompressed-size limiting is less explicit, and this quirk is Git-protocol-specific — it belongs at the Git boundary where someone reading the handler can see why it exists. It also keeps error responses in pkt-line format, which middleware wouldn't know to do.

**4. Regression tests in `api/src/tests/git_http.rs`.** Three cases, following the existing request-building patterns there:
